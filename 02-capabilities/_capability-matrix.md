## What the matrix reveals

- **Convergence — universal capability set is now narrow.** All five vendors deliver: shadow-IT discovery + risk score, sanctioned-app tagging, inline DLP (where they can decrypt), API DLP on a defined list of sanctioned apps, OAuth-grant inventory against the connected IdP, UEBA with impossible-travel, malware scanning of SaaS storage, share-link governance, GenAI category + inline prompt DLP. Below this floor, vendors look identical in an RFP cell-by-cell tick — which is exactly the trap, because the qualifiers matter more than the ticks.

- **Biggest divergence is deployment-mode posture, and it lines up with vendor lineage.** MDA is IdP-bound (no agent, no PAC, no forward proxy — CAAC reverse-proxy-after-handoff only); Netskope offers the widest steering set (agent, PAC, IPSec/GRE, IdP-routed SAML FP, SAML RP, RaaS); Palo Alto routes through GlobalProtect + Prisma Access, no reverse proxy; Skyhigh is the only one with mature SAML reverse proxy still positioned as first-class (but lens reviewers flag it as "on life support" for native/mobile in 2026); Zscaler is forward-proxy-only with CBI as the reverse-proxy substitute. The user's choice of *which deployment mode they need* (BYOD unmanaged-device control vs. managed-endpoint inline) collapses the field.

- **Encryption / tokenisation at gateway is effectively dead across the field.** MDA, Palo Alto, Zscaler explicitly do not provide it. Netskope inherited Kindite but flags it as "heavy lift, not the common path." Skyhigh routes through Ionic/Seclore DRM partner — not a native gateway feature. The original 2014-era CASB pillar has been deprecated industry-wide; practitioners pursuing it should look outside the CASB market.

- **Native session-kill / step-up MFA is also broadly absent.** Only Skyhigh's Cloud Access Policies expose step-up as a first-class verb. MDA's is in **preview**; Netskope/Zscaler/Palo Alto all route through the IdP (Entra CA / Okta) — meaning the response chain is: CASB detection → risk-signal API → IdP CA evaluation → next token refresh. There is no live mid-session step-up in 4 of 5 products (per netskope-product and zscaler-product lens reviews). Practitioners building runbooks must understand the IdP is the response plane, not the CASB.

- **Microsoft DfCA (user's primary case) — relative advantages.** First-class M365 connector depth (governance actions, granular activity policies that no other vendor matches for M365); tight Defender XDR + Entra ID + Purview integration (one identity, one classification engine, one incident graph); GenAI prompt visibility via DSPM-for-AI bound to MDE endpoints — uniquely tight for Microsoft-stack customers. CAAC sees the full set of in-session activities on a browser (block-print, block-Teams-send-message) that approaches Skyhigh/Netskope RP fidelity.

- **Microsoft DfCA — relative disadvantages.** No forward proxy, no agent, no PAC — IdP-bound by design (per MDA-architect lens review: "not a CASB peer to Netskope/Zscaler — an Entra-bound API+IdP-redirect overlay"). Coverage drops to API + Shadow IT discovery without Entra ID P1. Hard product ceilings (50 file policies/tenant, 200k/100k activity-policy auto-disable thresholds, 24–72hr Power BI/D365 log delay) bite at BFSI scale. No native watermark. Activity policies on non-M365 connectors have visibly thinner action sets. For non-Microsoft-stack workloads (Salesforce Mobile cert-pinning, Google Workspace externally-owned files unscannable, AWS/GCP only via resource activity not data) the product narrows considerably.

- **API-mode latency is the unspoken compliance fault line.** All five drafts label API DLP "Supported," but only Netskope's own copy is candid ("after-the-fact… not proactive"). Lens reviewers across all five vendors raised the same gap: regulator breach clocks (PDPA 72hr, RMiT supervisory notification, GDPR 72hr, MAS TRM 1hr) start at incident detection — and API mode adds minutes to hours to that clock. None of the vendors documents per-policy MTTR. This is the single largest gap in the whole field, not a vendor-specific weakness.

- **Per-app feature parity is asymmetric and Microsoft 365 wins everywhere.** Every vendor's API connector list has M365 as the deepest-supported app. The long-tail SaaS (regional fintechs, vertical-industry SaaS, internal portals) has *no* API coverage anywhere — inline-only, and inline only when SSL inspection holds. For the user's BFSI / Malaysian regulatory frame, this means the CASB decision is really a M365-control decision; anything outside that perimeter is uniformly under-served.

- **Lens-review verdict was unanimous: none of the 5 drafts is promotion-ready.** Architect-tier reviewers flagged every draft for missing perf data, missing BYOD/cert-pinning detail, vendor-marketing-grade catalogue numbers (Skyhigh 50k+, Netskope 1,800+, Zscaler 20k+, MDA 31k–34k drift) parroted as fact, and absent control-plane data-residency stories. Compliance-tier reviewers flagged every draft for missing control-framework mapping (BNM RMiT, MAS TRM, ISO 27017/27018, NIST CSF 2.0) and missing log-artefact / retention / immutability statements. Any cell above marked `[unverified]` should be re-validated against a current vendor SE walk-through before being relied on in a customer-facing artefact.
