# ISO/IEC 27018:2019 — CASB control mapping (framework page)

> Code of practice for protection of personally identifiable information (PII) in public clouds acting as PII processors. Where ISO 27017 is the general cloud-controls overlay, 27018 is the PII-processor-specific layer.

## Status

Framework page; full clause mapping is a separate deliverable. **Material is illustrative; not legal/regulatory advice.**

## Why ISO 27018 matters for CASB

ISO 27018 codifies the cloud-PII-processor obligations:
- Consent and choice — A.1
- Disclosure of sub-processors and law-enforcement requests — A.5
- Transparency — A.9
- Use, retention and disclosure limitation — A.10
- PII processor obligations — A.11
- Information security — A.12

For a regulated FI in MY / SG / HK / EU, the CASB vendor sits as a PII processor when:
- Activity logs contain user identifiers (always)
- DLP-matching content snippets are stored (regulated under PCI / equivalent if cardholder data)
- DSPM-for-AI prompt content is captured (personal-data treatment under GDPR)
- The CASB tenant data resides in a region governed by 27018-aligned obligations

## Key A.* clauses with CASB relevance

> **Verify against the published ISO/IEC 27018:2019 standard.** This is orientation.

| Clause | Subject (paraphrased) | CASB relevance |
|---|---|---|
| **A.1** | Consent and choice for PII processing | The lawful basis for CASB inspection of workforce traffic; documented workforce notice |
| **A.5.1** | Disclosure of sub-processors | The CASB vendor's published sub-processor list; change-notification SLA |
| **A.5.2** | Law-enforcement request disclosure | The CASB vendor's transparency report; processes for government / regulator request handling |
| **A.9.1** | Notice to PII principals | Workforce notice about CASB monitoring — included in Acceptable Use Policy / employee handbook |
| **A.10.1** | Use, retention, and disclosure limitation | CASB log retention policy; DLP incident-snippet handling |
| **A.11.1** | Disclosure of sub-processors and notification | Same as A.5.1 — sub-processor list and notification SLA |
| **A.11.2** | Public cloud PII processor's customer agreement | The CASB customer contract; data-processing agreement (DPA) |
| **A.12.x** | Information security controls overlay for PII | Encryption at rest, key management, access controls for processor systems |

## What practitioners need from the CASB vendor

For ISO 27018 evidence:

| Artefact | Where to get it |
|---|---|
| Vendor's ISO 27018 attestation / certificate scope | Vendor's Trust Center (Microsoft Service Trust Portal, Netskope Trust Center, etc.) |
| Sub-processor list | Vendor's published list — typically online with a feed for change notifications |
| Sub-processor notification SLA | Documented in the customer contract (typically 6-month advance notification) |
| Data Processing Agreement (DPA) | Customer-vendor contract; references 27018 obligations directly |
| Transparency report (law-enforcement requests) | Vendor's annual transparency report |
| Sub-region / sub-processor data flow diagram | Sometimes available; often only on request |
| Customer Lockbox / customer-controlled admin access | For Microsoft: Customer Lockbox feature documentation |

Without these, the audit conversation under 27018 is incomplete.

## Citation discipline

| Requirement | Detail |
|---|---|
| Cite clause + version | "ISO/IEC 27018:2019 A.11.1" |
| Distinguish 27018 obligations from 27017 (general cloud) and 27701 (PIMS) | Each has a distinct purpose; mixing them weakens audit traceability |
| Cite vendor attestation with report date and scope | "Microsoft's ISO 27018 attestation report dated YYYY-MM, scope including Microsoft Defender for Cloud Apps" |

## What an auditor will probe

| Question | Evidence required |
|---|---|
| Is the CASB vendor a PII processor under the contract? | DPA clause |
| Does the vendor have ISO 27018 attestation in scope for the product? | Trust Center attestation document |
| Sub-processor list — is the customer informed of changes? | Subscription to vendor notification feed; documented notification SLA |
| Where is regulated PII stored? | Tenant primary-data region; sub-processor data-flow path |
| What is the retention / deletion behaviour on contract exit? | DPA clauses + documented exit process |
| For DSPM-for-AI prompt capture: is the prompt content treated as PII? | Retention + access control documentation |
| For law-enforcement requests, what is the customer-notification process? | Vendor transparency policy |

## Composition with PDPA / GDPR / equivalent

ISO 27018 is a code-of-practice standard. It does NOT supersede national PII / data-protection law. For a Malaysian tenant:

- **PDPA Malaysia 2010 (amended 2024 — Act A1709)** is the binding national law; verify against gazetted text
- **ISO 27018** provides operational guidance and is often the customer-vendor contract reference
- **GDPR (if EU footprint)** is binding for EU personal data; 27018 is supportive evidence

Cite both the national law and the standard; do not substitute one for the other.

## Cross-border transfer

For 27018 A.11.1 and A.11.2:
- The CASB vendor's primary-data region determines where regulated PII (activity logs, DLP snippets, prompt content) resides
- Cross-border transfer requires documented basis (Standard Contractual Clauses for GDPR; PDPA's consent-or-transfer-mechanism for MY; etc.)
- Microsoft / vendor's transfer mechanism is documented in the DPA

For Microsoft Defender for Cloud Apps specifically: tenant primary-data region (verify against Trust Center — EU / US / UK / AU / Japan East as of 2025 H2). For Malaysian / Singaporean tenants, Japan East availability changed the residency conversation materially.
