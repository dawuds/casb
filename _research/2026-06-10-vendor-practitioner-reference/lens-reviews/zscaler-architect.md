# Architect lens review — Zscaler Internet Access (ZIA) with CASB / Data Protection

Reviewer posture: BFSI architect, MY-regulated, hybrid M365 + Workspace, Entra primary IdP with Okta legacy, mixed managed + BYOD. Read with scepticism. Source corpus is **search-snippet-grade vendor copy**; treat the whole draft as "not yet corroborated".

## Findings to act on (highest impact first)

1. **Entire draft sourced from vendor marketing + snippet text** — Section "Sources" (lines 270–309) lists 30 entries; 28 are `help.zscaler.com`, `zscaler.com/innovations`, or `zscaler.com/blogs`. Only two are non-vendor (`pulumi.com` registry, `securityscientist.net` blog). The author flags this in the "General uncertainty" block (lines 316–320) but the body of the doc still presents capabilities as "**Supported**" with vendor-controlled snippet citations. For BFSI promotion, every "Supported" claim needs at least one corroborator: an independent deployment guide, a partner SE-written blog, or a community.zscaler.com thread. Right now this doc cannot survive an RFP defence.

2. **SSL inspection prerequisite buried, not surfaced as a deployment gate** — Lines 26, 51, 85, 96, 109, 157 each restate "SSL inspection required". For BFSI this is the single biggest deployment cost (PKI, endpoint trust roll-out, bypass-list governance, regulator questions about decryption of personal banking sessions). Draft does not call out: (a) the operational cost of distributing the Zscaler intermediate CA to managed Win/macOS via MDM and the impossibility of doing so cleanly on personal BYOD without MDM enrolment, (b) the regulator-facing question of decrypting traffic on personally-owned devices in MY under PDPA, (c) the standard 15–20% TLS-inspection bypass list that any real BFSI tenant ends up with (banking portals, broker apps, gov.my services, healthcare apps) and how those bypasses **silently disable** all CASB/DLP for those destinations. Add a "SSL inspection footprint" subsection. Quantify the bypass-list growth as a real risk, not a footnote.

3. **Identity dependency is hand-waved** — Line 62 says "Step-up auth / MFA is not a Zscaler primitive — Zscaler steers, the IdP authenticates." That is correct but incomplete. Draft does not say: (a) ZIA user attribution requires SAML/SCIM from Entra (or the Okta legacy), (b) inline policy that filters by user group only works after SCIM provisioning completes — net-new joiners get default policy until sync, (c) for the hybrid Entra + Okta legacy estate the BFSI is running, **which IdP carries which user population matters** because Cloud App Control filters resolve against whichever SCIM feed Zscaler is bound to. Tenant Restriction policy (#3, lines 89–98) is built on Entra tenant GUID — useless if the legacy Okta-fronted population accesses M365 via a different auth path. Surface this.

4. **Client Connector footprint is not quantified for BYOD** — Line 24 lists it as "the default for managed endpoints". Line 41 admits "Devices that cannot install the Client Connector ... are blind spots." For the BFSI BYOD population (iOS + Android personal) this is the **majority case** for SaaS access, not the edge case. Zscaler Client Connector for iOS/Android requires MDM-pushed VPN profile + the user accepting an on-device VPN — neither is realistic on personal devices in MY without an explicit corporate-managed-app posture (Intune App Protection or equivalent). Draft does not enumerate: agent install vector per OS (MSI/PKG/MDM/App Store), per-platform on-device permissions required, the user-experience friction of an always-on VPN icon on a personal phone, and the regulator-facing question of whether the bank can compel agent install on a personally-owned device.

5. **"Reverse proxy = CBI redirect" substitution is presented as equivalent — it is not** — Line 27 and line 259 both say the Zscaler equivalent of a Netskope/MDCA SAML-rewrite reverse-proxy is "agentless Cloud Browser Isolation triggered via IdP". This is the most important architectural difference for a BFSI buying decision and the draft files it under "different UX, not a 1:1 substitute" without naming the consequences. CBI ships pixels: no native client functionality, no Outlook desktop, no Teams desktop client, no OneDrive sync client, **no mobile Outlook on iOS/Android**. For a BYOD population that expects to use the M365 mobile apps natively, CBI-as-reverse-proxy means "your personal phone gets a remote browser tab, not the Outlook app". That is a usability cliff. Promote this from buried caveat to a named architectural limitation.

6. **Inline DLP false-positive cost understated** — Policy #4 (line 110) acknowledges "High" FP risk for inline DLP. What the draft does not say: in BFSI the cost of an FP-blocked legitimate upload to a regulator portal (BNM, SC) or an external auditor's secure portal is incident-grade. Add operational guidance: stage all inline DLP rules in `Allow + Log` for at least 30 days; `Confirm (user-justification)` action is the production default for anything that touches business-critical flows, not `Block`. The draft lists `Confirm` as one of four actions on line 108 but does not advise treating it as the production default.

7. **Latency / TLS-inspection compute is not quantified at all** — Section "Capability set" lines 49–63 lists features with no performance commentary. For a BFSI promotion the architect needs at least order-of-magnitude: median added RTT through a ZIA service edge in KUL / SIN region, throughput cap per Client Connector tunnel, behaviour under sustained TLS-inspection load (e.g. large file uploads to Box). The draft is silent. If this cannot be sourced from non-vendor material, say so explicitly. Do not let the architect lens promote a doc that has zero perf data.

8. **"Off-Zscaler = invisible" is the structural truth — say it once, loudly** — Repeated across lines 39, 41, 257, 268. Consolidate into a single architectural caveat at the top of the doc: **everything ZIA's inline plane sees depends on traffic steering. Anything off-steering — BYOD without agent, contractor laptop, kiosk, IoT, agent-disabled-by-user, split-tunnel exclusion — is dark to inline CASB/DLP/Cloud-App-Control/Shadow-IT.** This is the single most important architectural property of the product and it is currently scattered across five bullets.

9. **Overlap with FWaaS / SWG / ZTNA inside the same SSE is not addressed** — The brief asks where this sits alongside SWG / ZTNA / FWaaS / DLP-at-rest. Draft does not say. ZIA itself is SWG + FWaaS + inline CASB collapsed into one proxy — that is the marketing pitch and also the operational risk (line 313 surfaces this as a reviewer prompt but does not resolve it). Add a section: "ZIA collapses SWG + FWaaS + Cloud App Control + inline DLP into one TLS-inspecting forward proxy. ZPA is the separate ZTNA plane. SaaS Security API is the only non-inline plane. DLP-at-rest for M365 / Workspace is duplicated against Microsoft Purview / Google Drive DLP and against Defender for Cloud Apps; the BFSI must pick a primary DLP plane to avoid double-quarantine races." Name the duplication.

10. **Certificate-pinned apps treatment is mechanical, not operational** — Line 258 names cert-pinning as a documented limitation. Draft does not enumerate which BFSI-relevant apps actually pin: WhatsApp, Signal, mobile banking apps generally, some Microsoft mobile clients historically, the iOS Mail app for some Exchange paths. The practitioner-relevant question is not "does Zscaler handle pinning" (it doesn't, by design) but "what fraction of BYOD-relevant traffic ends up on the SSL bypass list because of pinning". Quantify or admit you cannot.

11. **API-only coverage gap on third-party-to-third-party traffic** — Brief asks about third-party-to-third-party API traffic (e.g. SaaS-to-SaaS data flows that never touch the user's endpoint: Salesforce → Slack via native integration, M365 → ServiceNow via Graph). Draft is silent. 3rd-Party App Governance (line 53) catches OAuth grants but not the runtime data flow. SaaS Security API DLP catches data **at rest** in a connected tenant but not the in-flight movement between two SaaS tenants. State this explicitly.

12. **"Encrypted by user" content gap is missing** — Brief asks. Draft mentions E2EE chat (line 263) but not user-applied encryption (e.g. a user PGP-encrypting a file before uploading to OneDrive, or zipping with a password before sending). Inline DLP on the body sees the ciphertext and matches nothing; API DLP sees the same. Standard practitioner trick to defeat DLP. State it.

13. **Licensing claims hedged but still risky** — Lines 14–17 list ZIA Essentials / Business / Advanced/Transformation with `[unverified for current 2026 SKU sheet]` and `[snippet]` flags. The author has done the right thing by flagging. Before promotion, get the 2026-current Zscaler price book from a Zscaler SE under NDA and re-write this section against it. Do not let `[unverified]` SKU claims into a customer-facing artefact. Today's date is 2026-06-10 — the 2024-vintage tier names are now two cycles old.

14. **3rd-Party App Governance lineage caveat** — Line 10 says "Canonic Security acquired 2023 → became 3rd-Party App Governance / 'AppTotal'". The "AppTotal" name predates the Canonic acquisition as Zscaler's own labelling for SaaS-supply-chain risk. Verify whether AppTotal was a Zscaler-internal product folded with Canonic, or whether Canonic *was* AppTotal. Get the acquisition release. Minor but signals due-diligence quality.

15. **No incident / outage history surfaced** — A real BFSI architect-lens review names the operational risk of a single-vendor SSE chokepoint. Zscaler has had service-edge incidents that broke entire customer estates' internet access. None of that is in the draft. At minimum, add a "Concentration risk" caveat: ZIA outage = simultaneous loss of SWG + FWaaS + inline CASB + inline DLP + Shadow-IT enforcement. SaaS Security API plane survives an inline outage but inline does not survive an API plane outage either way — they are independent failure domains. State this. Cite community-reported outages if a non-vendor source is reachable.

- ApplicationRisk Index 1–5 scale and the per-app risk attributes (lines 49–50) — only snippet-cited.
- Native API connector list (lines 28–38) — every entry is `[snippet]` or `[unverified]`. Read the current SaaS Security API help-portal index end-to-end and produce the authoritative list with API-mode-DLP / Malware / SSPM coverage per app on a matrix.
- WebSocket inspection for Copilot-on-web (line 63 and line 157) — single blog post citation. Confirm the feature is GA, not a roadmap item, and what licence tier it lands in.
- Quarantine remediation specifics for Box / OneDrive / SharePoint with tombstone template, Salesforce "Restore" (line 52 and line 119) — innovation-announcement-grade, not GA-doc-grade.
- "Brute force, impossible travel, bulk uploads, off-hours, encrypted uploads" as UEBA primitives (line 54) — snippet-only. Confirm which of these are out-of-box vs configurable thresholds.
- 3rd-Party App Governance ingesting OAuth grants for M365 / Workspace / Salesforce / Slack / GitHub (line 53) — confirm the per-tenant scope coverage and the latency between a new OAuth grant in M365 and the alert in Zscaler.
- SSPM compliance mappings to CIS / NIST 800-53 (line 194 flagged `[unverified]`) — verify before letting the compliance lens cite this.
- ALLOW / BLOCK / CAUTION / ISOLATE action set (line 83) is corroborated by the Pulumi registry (line 308). Good — keep that citation, it is the strongest non-vendor reference in the doc.

## Policies to refine

- **Policy #2 Cloud App Control — Block High-Risk Unsanctioned Apps**: add a default-deny vs default-allow framing. BFSI buyers will ask whether the rule order means "deny-listed apps blocked, all others allowed" or vice versa. Draft does not specify the default action for traffic that matches no rule.
- **Policy #3 Tenant Restriction**: name the M365 personal-account collision risk explicitly (line 97 hints at it) — when a corporate user signs into `login.microsoftonline.com` with a personal `@outlook.com` identity, Tenant Restriction needs the Restrict-Access-To-Tenants header set with the corporate Entra tenant GUID, and the equivalent for Google Workspace uses `X-GoogApps-Allowed-Domains`. Both header injections happen at the ZIA proxy and require SSL inspection on those specific hosts. State this.
- **Policy #4 Inline DLP for SaaS Upload**: change recommended default action from `Block` to `Confirm (user-justification)` with `Allow + Log` for a 30-day baseline before any `Block`. Current draft lists actions but does not stage them.
- **Policy #5 SaaS Security API DLP — Data at Rest**: name the first-scan storm. When the API connector is first enabled against a tenant with N TB of existing OneDrive/SharePoint content, the initial scan walks the entire content store and generates a finding flood. Stage with the rule disabled, baseline the finding rate against a small OU, then enable broadly. Draft does not mention this.
- **Policy #8 AI & ML Application — Granular Activity Control**: the granular activities list (`chat / upload / download / share / delete / invite` on line 154) is ChatGPT-specific. For Copilot the activities are different (line 63: `Chat / Delete / Rename / Share / Upload`). Generalise the policy framing — per-app activity sets vary, do not present one app's set as the schema.
- **Policy #9 Cloud Browser Isolation**: name the UX cliff. CBI breaks drag-drop file upload (line 169 hints at it), breaks browser-extension authenticators, breaks SSO bookmarklets, breaks anything that needs native clipboard, breaks anything that uses WebRTC for voice/video. For BFSI execs who routinely paste from Excel into a SaaS form, watermark + clipboard-out-block is a hard friction point. Name the user-class impact, not just "graphics-heavy apps".
- **Policy #10 3rd-Party App Governance — OAuth Grant Policy**: name the time-to-detect. OAuth grants made via admin consent are visible quickly; user-consent grants depend on the SaaS tenant audit-log polling cadence. Without a number the policy is unactionable.
- **Policy #11 SSPM**: add a remediation-blast-radius caveat. Auto-remediation `on` against a tenant-wide setting drift can revert a setting an admin intentionally changed for an audit. Auto-remediation `off` is the BFSI default; tickets-and-approval flow goes via the SOAR.

## Limitations the draft missed

- **Decryption posture and PDPA/RMiT**: the draft does not name the regulatory question of whether a Malaysian regulated FS can decrypt employee traffic on personally-owned devices under PDPA, or whether the bank's acceptable-use policy needs explicit consent for TLS interception. This is a real deployment blocker for BYOD steering. (Not legal advice. The architect lens should at least name the question, not skip it.)
- **Mobile native app coverage gap**: cert pinning is named; the broader point that Zscaler Client Connector on iOS routes via per-app VPN or device-wide VPN, and that **mobile native SaaS apps that pin will be on the bypass list by necessity**, is not stated. Net effect: inline CASB/DLP on the BYOD mobile estate is largely a fiction unless the bank pushes managed-app posture via Intune App Protection.
- **User-applied encryption / archive password / steganography**: standard DLP-bypass tradecraft, unmentioned.
- **SaaS-to-SaaS native integrations**: Slack ↔ Salesforce, M365 ↔ ServiceNow, Box ↔ Slack — runtime data flows that never traverse the user endpoint. Inline plane blind. API plane catches data-at-rest only after the fact. State.
- **Endpoint DLP for clipboard / screenshot / USB**: line 264 names this. Good. Strengthen: name that Zscaler's Endpoint DLP is a separate product with separate licensing; this draft is the network+SaaS plane only.
- **Single-tenant-fronted multi-region SaaS**: BFSI buyers will ask whether ZIA can enforce policy on a SaaS app that fronts traffic via a single global CDN POP (Cloudflare-fronted, Akamai-fronted) where the destination IP gives no per-tenant signal. The draft is silent. This is real — Tenant Restriction headers help on M365/Workspace because those vendors publish the headers; for SaaS apps without such headers there is no per-tenant separation.
- **Concentration risk on a single SSE plane**: stated in finding 15 above.
- **Telemetry export / SIEM integration**: BFSI buyers will ask about NSS / Log Streaming Service / Cloud NSS to the SIEM (Splunk, Sentinel, Chronicle). Not in the draft. Without this the SOC has no view of the CASB events.
- **Audit-log retention on ZIA itself**: standard ZIA retains short-window logs; deeper retention requires NSS-to-SIEM or LSS-to-cloud-storage. Not in the draft. BNM RMiT will ask.
- **Change-management cadence**: Zscaler service edge ships behaviour changes more frequently than a customer-controlled appliance estate. Not mentioned. Architect lens should at least name "vendor controls the data plane; customer change-window is partial".

## Overall verdict

- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** Snippet-grade vendor sources only, zero perf/latency data, BYOD and BFSI deployment gates buried as footnotes — needs corroboration and a deployment-footprint section before it can survive an architect-tier defence.
