# Architect lens review — Palo Alto Prisma Access (SaaS Security / Inline CASB + Aperture API)

Reviewer stance: senior architect, BFSI Malaysia, hybrid M365 + Workspace estate, Entra primary IdP + Okta legacy, managed Win/macOS + BYOD iOS/Android. Read with the assumption that every "Supported" claim hides a deployment cost the draft did not price.

## Findings to act on (highest impact first)

1. **Identity dependency is asserted but not architected.** Lines 78, 97-98, 104-108, 113, 145. Draft repeatedly references "AD/Entra ID group", "HIP", "Sanctioned tenant ID", but never states the IdP integration topology that Prisma Access actually needs: SAML/SCIM from Entra to Prisma Access Cloud Identity Engine, group-mapping refresh cadence, and the fact that GlobalProtect user-ID resolution is what carries the group claim downstream. In a real Entra-primary + Okta-legacy estate this is a non-trivial integration with two failure modes (CIE outage = policy fail-open to default action; stale group sync = wrong policy applied to an employee mid-onboarding). Add an "Identity prerequisites" subsection or this gets shipped as a half-spec.

2. **Traffic steering claim hides the BYOD gap entirely.** Lines 27-33. The draft lists GlobalProtect, IPSec/GRE, Explicit Proxy, Clientless VPN as steering options, then quietly notes Clientless is "limited to web apps … subset of sanctioned SaaS." In a BFSI estate with personal iOS/Android, that is the *only* relevant steering option for BYOD, and it does not carry native SaaS clients (Outlook Mobile, Teams Mobile, OneDrive Mobile, Drive iOS, Workspace iOS). The draft never resolves what *actually* happens to BYOD mobile native-app traffic — answer: it bypasses inline entirely and you are left with API-side controls only. This is a deployment gap dressed as a feature list. Call it out plainly.

3. **SSL inspection cert PKI is invisible in the draft.** Decryption is named as "required" four times (44, 48, 66, 82, 106, 154) without a single line on the PKI choreography. Practitioner needs: Prisma Access intermediate CA chain or BYO-CA; distribution to managed endpoints (Intune/Jamf MDM profile push); pinning bypass list for known incompatible apps (Apple Push, Google FCM, banking apps, MFA apps, Microsoft Defender XDR client telemetry, native EDR sensors); exclusion of regulator-sensitive traffic (NACSA, BNM portals) from inspection. Draft does not enumerate. Without this section any BFSI deployment will brick at least one critical client app in week 1.

4. **Two-console policy seam is correctly flagged but underweighted.** Lines 38-40, 192. Draft calls this "the priority note" but does not state the operational cost: the *same* control objective (block external sharing of PCI) requires two policy authors maintaining two policy graphs across two consoles using two different policy DSLs, with a shared Enterprise DLP profile as the only common artifact. That is not a "seam" — that is a permanent doubled change-management overhead, audit-trail fork, and rule-drift risk. For a BNM RMiT-regulated estate this is a control-design problem (one logical control implemented twice = two attestations, two break-glass paths). Promote this from a footnote to a top-three architectural concern.

5. **Performance / latency / TLS inspection compute is completely absent.** Draft does not contain the words latency, throughput, RTT, decryption capacity, or PoP location in any operational sense. For Malaysian users the relevant PoPs are SG / KL (if available) / Jakarta; transit costs to US-region SaaS tenants add 150-250 ms baseline before inspection. TLS decrypt + DLP body inspection + Enterprise DLP regex evaluation adds further. Vendor docs do not publish this; practitioner needs to size during PoC. Add a "Performance / sizing unknowns" subsection — currently the draft promotes the product as if it were free of compute cost.

6. **Where Prisma Access sits alongside SWG/ZTNA/FWaaS/DLP-at-rest is conflated.** Draft treats Prisma Access as a monolith. In reality: SWG (URL filtering, threat prevention) and inline CASB share the decryption fabric and policy engine; ZTNA (Prisma Access ZTNA Connector / Cloud Identity Engine) is a separate policy plane with its own connector topology; FWaaS is the same decryption fabric but L3/L4 policy. If a BFSI shop already runs Zscaler ZIA / ZPA, Netskope, or Microsoft Global Secure Access, deploying Prisma Access inline CASB drags the whole SSE platform with it — you do not buy "just the CASB". The draft does not state this trade. Add it.

7. **"Real-time session-kill is not native" is correct but understated.** Lines 56, 200. For BFSI incident response the absence of native session termination is a material control gap: detection at T+0, XSOAR runbook + IdP revoke at T+minutes-to-hours. State this as a *control coverage* limitation, not a feature footnote. NIST SP 800-61r3 IR lifecycle and BNM RMiT 11.18 expectations on containment timing make this consequential.

8. **Encrypted-by-user content gap is missing.** Draft does not address user-side encryption (1Password vaults, Cryptomator containers in Drive, password-protected zips, S/MIME / PGP mail). API CASB scans plaintext; user-encrypted blobs evade DLP entirely. This is a known CASB-wide limitation but the draft is silent.

9. **Forward-scan-only apps for several Tier-1 surfaces is a coverage hole, not a footnote.** Line 50, 195. Gmail / Google Chat / ChatGPT Enterprise being forward-scan-only means *zero* historical at-rest scanning for an M365-or-Workspace estate's most prolific data surface. For a BFSI shop migrating to Prisma Access with 5+ years of Gmail/Drive history, the practitioner assumption "API CASB scans my tenant" is wrong on day one. Promote to architectural finding.

10. **Agent rollout footprint is not sized.** GlobalProtect on Win/macOS/iOS/Android/Linux/ChromeOS — the draft lists it as a steering option without any operational footprint: MDM packaging, certificate enrolment, captive-portal handling on hotel/cafe WiFi, Always-On vs On-Demand profile choice, conflict with Windows Always-On VPN, conflict with macOS Network Extension limits (only one full-tunnel NE can be active — collides with EDR network filters, Cloudflare WARP, Zscaler Client Connector). BFSI shops typically already have 2-3 network agents on endpoints; GlobalProtect must coexist. Draft is silent.

## Capability claims to corroborate (do not promote without independent source)

- **"~28 sanctioned apps" with API connectors as of mid-2026 (line 35).** Vendor doc only. Cross-check against the live supported-apps matrix at promotion time; this number drifts quarterly.
- **"55+ of 80+ tracked attributes per app" risk score (line 44).** Vendor doc only — no third-party validation of the risk-score methodology. Treat as black-box.
- **"Behavior Threats uses up to 90 days of historical data for ML baselines" (line 54).** Vendor doc only. No external benchmark on detection efficacy or false-positive rates.
- **"WildFire-backed scanning of files at rest" (line 62).** Vendor doc. Independent comparisons against MS Defender / Google native malware scan would be useful but were not cited.
- **Claude / Gemini tenant-header enforcement (line 146).** Draft itself flags this as needing per-app verification — good. Do not promote until Anthropic and Google publish enterprise-account header schemas with stable IDs; today's headers can be deprecated without notice.
- **"25 MB file ceiling … hard limit across all apps" (line 50).** Vendor doc; draft itself flags possible support-gated increases. Treat as accurate today, but verify against the live matrix.
- **"No public independent CVE / breach record" (line 230).** Draft author flagged not exhausting this. Architect lens position: assume there are CVEs in GlobalProtect specifically (well-known target — CVE-2024-3400 series). Distinguish "Prisma Access SaaS Security backend CVEs" from "GlobalProtect client CVEs" — the latter exist and matter.
- **"Microsoft Learn corroborates tenant-restriction header insertion" (line 219).** This is the *only* independent source in the whole draft. Everything else is vendor. State this plainly — single independent source does not validate the rest of the surface.

## Policies to refine

- **Policy #1 (Block PCI/PII upload to unsanctioned GenAI):** Add a precondition that AI Access Security license is active (separate SKU per line 24). Add the "what if GlobalProtect not running" failure mode — fail-open to default rule, which on BYOD means traffic exits via cellular and never sees the policy.
- **Policy #2 (Quarantine externally-shared Drive PII):** Add the inverse policy on the inline side, or state explicitly that this is API-only and historical files only — net-new uploads from the same user the same day go through inline and require Policy #1's twin. This is the two-console seam in operation. The worked-example YAML should make this explicit (e.g. `companion_inline_policy_required: true`).
- **Policy #3 (Block SharePoint download to unmanaged):** State the BYOD trap — HIP requires GlobalProtect, GlobalProtect on iOS/Android is full-tunnel and operationally heavy for BYOD. Most BFSI shops do not deploy GP on personal mobile. Therefore this policy *does not apply* to the BYOD mobile native-app case where it is most needed. Either name an alternative (Conditional Access App Control via Entra — but that is Microsoft's reverse proxy, which Palo Alto explicitly does not have) or admit the gap.
- **Policy #4 (M365 tenant restriction):** Add HTTP Header Insertion's interaction with TLS decrypt — header insertion *requires* decryption (you cannot inject headers into TLS you have not terminated). Add the Entra reciprocal config — Conditional Access policies need to expect the header. Cross-link to MAS / HKMA equivalents if this is positioned for BFSI.
- **Policy #5 (Impossible travel on Salesforce):** Static thresholds are weak. State the corporate-VPN false-positive at non-trivial volume, particularly for users on Prisma Access PoPs in SG/HK serving Malaysian users — that *is* impossible-travel from Salesforce's view if a user roams between PoPs. Mitigation needs a Prisma Access PoP egress IP allow-list, not just "corporate VPN IPs".
- **Policy #6 (Revoke high-risk OAuth in Workspace):** Auto-revoke on a tier-1 grant is operationally dangerous. State explicitly: pilot in alert-only for 30 days, build the approved-vendor allow-list first, then convert. The draft says "needs an allow-list of approved vendors first" but does not lock auto-revoke behind it.
- **Policy #7 (Bulk download Box off-hours):** State that "user-local" time requires accurate Identity time-zone attributes — not a given for cross-border BFSI staff. Default time-zone fallback is a known source of UEBA false-positives.
- **Policy #8 (Malware scan OneDrive uploads):** State that this duplicates Microsoft Defender for Office 365 + Defender for Endpoint cloud-detection on the same tenant. Two scanners on the same file is operationally fine but pricing-wise wasteful unless deliberately defence-in-depth.
- **Policy #9 (Sanctioned-tenant control for ChatGPT/Claude/Gemini):** Already correctly hedged in the draft. Add: this is inline-only. Native iOS/Android ChatGPT / Claude / Gemini apps on BYOD bypass.
- **Policy #10 (Coach away from Tolerated file-sharing apps):** Add the metric — coach pages without a measurement loop (click-through, subsequent behaviour change) are security theatre. Specify what telemetry feeds back into the policy review cadence.

## Limitations the draft missed

- **BYOD mobile native apps (iOS/Android) bypass inline entirely.** Draft acknowledges cert pinning (line 197) but not the broader BYOD-without-MDM-managed-GlobalProtect case. Inline = nil. API CASB is the only control surface for BYOD. State plainly.
- **Unmanaged endpoints (contractors, M&A targets, third-party access) have no equivalent control plane.** Draft mentions Clientless VPN (line 33) as the unmanaged-access path but does not state its functional scope: web-app reverse-proxy for a vendor-curated list of sanctioned apps; no native client; no DLP body inspection on most apps.
- **Third-party-to-third-party API traffic is invisible.** Server-to-server SaaS integrations (Salesforce ↔ Workday, ServiceNow ↔ Slack, Workday ↔ Box) traverse no endpoint, no Prisma Access PoP. Neither inline nor API CASB covers this unless the destination app has an API connector and the integration writes scannable objects there. SSPM gives partial posture-level visibility; runtime data flow is uncovered.
- **Certificate-pinned apps in non-mobile contexts.** Draft scopes pinning to "mobile native" (line 197). Desktop apps also pin (some banking SaaS clients, MFA desktop tools, certain EDR clients, conferencing native clients). Same bypass dynamic.
- **User-encrypted content** (per Finding 8) — Cryptomator, Veracrypt containers, password-protected archives, S/MIME/PGP mail.
- **Watermarking / DRM and field-level tokenisation absence** is acknowledged (lines 198-199) but framed as "no native" rather than "if you need this you cannot use Prisma Access alone — you need an MIP / Purview / Virtru / Skyhigh-like overlay". State the architectural implication.
- **Egress from Prisma Access PoPs and SaaS tenant geo-pinning.** Draft mentions data-residency at PoP level (line 58) but not that egress from a Malaysian user via a Singapore PoP to a US-East SaaS tenant means data crosses three jurisdictions before scanning, with implications for PDPA cross-border-transfer assessments.
- **CASB log retention / forensics horizon.** Strata Logging Service retention is licensed (30 / 90 / 365 day tiers, additional charge for longer). Draft does not state. For BFSI evidentiary requirements (BNM RMiT 10.49 minimum 7-year audit-log retention for relevant logs), this is a sizing question, not a footnote.
- **Break-glass for inline outage.** No mention of what happens to user traffic when Prisma Access PoP is unreachable or when GlobalProtect is in tunnel-loss state. Fail-open / fail-closed behaviour matters for BFSI continuity planning.
- **EU AI Act / NIST AI RMF coverage for the GenAI policy stack** is not addressed; for Malaysian BFSI this is illustrative not binding, but architect should know AI Access Security's evidentiary outputs (audit trail of prompts blocked, content matched) need to land in the SIEM with sufficient fidelity to support model-risk attestations.

## Overall verdict

- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** Draft is well-structured and the policy examples are credible, but the deployment prerequisites (identity, PKI, agent footprint, performance, BYOD scope) are absent and the source mix is 13-of-14 vendor-controlled — promote only after a deployment-prerequisites section is added and at least three claims (~28 connectors, 25 MB ceiling, Behavior Threats baseline window) are corroborated against independent sources or live vendor matrices at promotion time.
