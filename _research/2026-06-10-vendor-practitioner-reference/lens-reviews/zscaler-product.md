# Product lens review — Zscaler Internet Access (ZIA) with CASB / Data Protection

Reviewer stance: hands-on CASB engineer who has administered ZIA-CASB in BFSI production. The draft is structurally honest about its snippet-grade sourcing, but it still parrots vendor language in places, and it omits operationally important footguns. Everything below assumes the draft will become a practitioner reference that a BFSI customer might actually use — not a marketing handout.

## Findings to act on (highest impact first)

1. **The whole document is built on JS-rendered help-portal pages that returned empty bodies.** The draft says so once and then proceeds to make 30+ specific feature claims sourced to those same blank pages. Every `[snippet]` marker is doing load-bearing work, and a practitioner reader will skim past them. Either (a) re-fetch with a headless browser before promotion, or (b) demote every snippet-only claim to "Vendor positioning — unverified at help-portal page level". Right now the document reads as "Supported" too freely. Affects lines 49–58, 75, 87, 98, 111, 123, 135, 147, 159, 170, 184, 197.

2. **"Inline DLP for sanctioned SaaS — Supported" (line 51) is materially overstated as written.** It is supported for traffic that (a) traverses ZIA, (b) is decrypted, (c) is not on a pinning-driven bypass list, AND (d) is one of the body types the inline DLP engine actually parses. The fourth condition is missing: ZIA's inline DLP has historically struggled with multipart-chunked uploads, resumable uploads (Google Drive resumable, OneDrive large-file sessions), and gRPC/QUIC-over-HTTP3. QUIC handling has been an ongoing footgun — many shops still block QUIC outright so DLP works. None of this appears in the draft. Reclassify to **Supported with caveats**, and name the upload-protocol limitation.

3. **"OOB CASB" remediation latency is glossed.** Line 42 says "minutes to hours depending on the SaaS app's webhook / change-feed", which is correct in direction but soft. In practice: M365 change notifications via Graph are typically 1–5 min steady-state but routinely degrade to 15–60 min under Microsoft-side throttling; Salesforce streaming uses a 24-hour event retention buffer that the practitioner has to design around; Box and Dropbox webhooks are near-real-time when healthy but offer no replay if Zscaler drops one. Replace the soft sentence with a per-connector table or at minimum name the M365 throttling reality. A BFSI customer expecting "near real-time" because the vendor said "continuous (change-feed)" will be unhappy.

4. **Tenant Restriction is presented as a clean primitive (Policy #3). It is not.** Three footguns are missing: (i) Microsoft's own *Tenant Restrictions v2* requires the **Restrict-Access-Context** header injection that needs SSL inspection AND a specific header-rewrite policy AND the Entra side configured — Zscaler alone does not deliver TR v2. The draft conflates Zscaler's classic Tenant Profiles with TR v2. (ii) Google Workspace tenant restriction relies on `X-GoogApps-Allowed-Domains` header injection, with its own quirks on consumer redirects. (iii) "Allow read-only (no upload)" for the Microsoft case depends on which Microsoft endpoint the client hits — Office on the web vs the Outlook desktop client vs the OneDrive sync client behave differently. Reviewer should split the Microsoft case from the generic case and call out TR v2 explicitly.

5. **Cloud Browser Isolation is sold harder than it deserves (Policy #9 and line 27).** CBI is real, but as a "reverse-proxy equivalent" it has known operational pain: (a) file-download policy via CBI does not give the practitioner the same DLP hooks as the inline data path — DLP is applied to the proxy ingress, not the CBI render — and large-file downloads have historically been awkward; (b) clipboard policy granularity differs between Windows, macOS, and ChromeOS endpoints; (c) the CBI session cost-per-user becomes material if you scope it to "all unsanctioned apps with risk ≥ 4" as the draft cheerfully suggests; (d) Single Sign-On into the isolated session has historically required extra IdP work — the draft implies seamless redirect. Demote line 27's "the equivalent on the Zscaler platform" to "a partial substitute" and name the constraints.

6. **"Step-up auth / MFA is not a Zscaler primitive" is the right verdict — but the draft does not name the integration that is.** Practitioners do this via Zscaler-to-Entra Conditional Access signals (risky session triggers Conditional Access re-auth) or via Okta workflows. The draft tells the reader Zscaler cannot, but does not tell the reader what to do instead. That is a half-finished limitation.

7. **GenAI / Copilot section parrots vendor blog copy (line 63).** "Prompt classification + inspection", "DLP applied to prompt bodies", "WebSocket inspection" — these are exactly the marketing bullets from the 2025 ZIA innovation launch. The practitioner-relevant question is *which* prompt fields are inspected: does Zscaler see the system prompt, the file attachments in ChatGPT Enterprise, the Copilot Graph-grounding payload (which is server-side and never on the wire)? The answer for Copilot M365 on the wire is "very limited — most of the grounding never leaves Microsoft" — Zscaler can inspect the user-typed prompt going *out* of the browser but not the retrieval-grounded context. The draft does not say this. A BFSI reader will assume Zscaler "sees" Copilot. It does not see what matters most.

8. **"Application Risk Index 1–5" (line 49) is reported uncritically.** The Index is vendor-curated, opaque in methodology, and changes silently. A practitioner who builds policies on `risk_index >= 4` discovers later that vendor reclassification moved an app and policies re-fire (or stop firing). Name it. The draft says "treat the number as marketing" about the catalogue size but not about the Index itself.

9. **API connector list is presented as fact (lines 29–38) but flagged as snippet.** That is the wrong combination. Either confirm by reading the SaaS Security API connectors index page in a real browser, or move the list to "vendor partner pages claim coverage of …". Webex API parity, in particular, has historically been read-mostly with limited remediation; the draft's "[unverified for current API parity]" is correct but should be promoted to the body.

10. **Shadow-IT report is described as if it were a CASB-grade discovery tool.** It is not. It is a log-mining report over ZIA's own traffic. The draft says this at line 39 and again at line 268, which is good, but Policy #1 ("Shadow-IT App Sanctioning + Risk Tagging") then describes it as a tagging primitive without restating the constraint. The practitioner reader who lands on Policy #1 first will not connect it to the limitation buried in line 39. Repeat the constraint inside Policy #1.

11. **The Configurable Policies section has no "policy precedence / rule order" discussion.** ZIA evaluates Cloud App Control rules in numbered order with first-match-wins semantics in most cases, but DLP, URL Filtering, and Cloud App Control overlap and the practitioner-relevant question — "if I block ChatGPT in URL Filtering, do my Cloud App Control granular activity rules ever fire?" — is unanswered. This is the single most common practitioner mistake on ZIA and the draft is silent on it.

12. **Worked example (lines 203–253) is a fabricated schema.** The draft admits this at line 201, but the JSONC keys (`granular_activities`, `device_trust_level`, `tenant_profile_ref`) do not match the Pulumi/Terraform provider's actual attribute names that the draft itself cites at line 308. Either align to the real provider schema or drop the example. Half-real, half-fictional schema is the worst of both worlds for a practitioner reference.

## Capability claims to corroborate (do not promote without independent source)

- **20,000+ application catalogue.** Pure vendor copy. Source against an independent app-discovery benchmark or strip the number.
- **EDM / IDM / OCR availability inside the inline data path** (line 51, line 106). EDM and IDM both have indexing-corpus prerequisites that the draft skips. OCR has historically been an "advanced DLP" add-on with size and language limits — neither named.
- **"Tombstone template" remediation for quarantine.** Line 119. The tombstone behaviour differs across Box / OneDrive / SharePoint and is not symmetric — Salesforce uses Restore semantics, the rest use replace-with-stub. Re-read the help page in a browser.
- **"WebSocket inspection" enabled for Copilot on the web.** Line 63 and Policy #8. This is vendor blog material from a 2025 innovation launch. Confirm against the help portal that it is GA in 2026 and not still gated to early access.
- **SSPM-to-CIS / NIST 800-53 compliance mapping.** Line 194, already flagged `[unverified]`. Do not promote without browser-verified screenshot of the mapping page.
- **"Auto-remediation: on / off (where the SaaS API supports write)"** in Policy #11. The "where the SaaS API supports write" qualifier is doing a lot of hidden work — list which connectors actually have write remediation vs read-only posture findings, or strip auto-remediation from the keynobs.
- **3rd-Party App Governance scope coverage.** Line 53 implies parity across M365 / Workspace / Salesforce / Slack / GitHub. Coverage depth varies materially — Salesforce connected-app governance is shallower than M365 app-consent governance. Source against the Canonic/AppTotal coverage matrix.
- **Avalor / Data Fabric for Security** (line 11) is mentioned but not used downstream. If it does not feed CASB risk surfacing in the practitioner workflow, drop the mention — it is currently noise.
- **"Source IP Anchoring" for unmanaged-device SaaS steering** (line 25). Real feature; not all SaaS apps cooperate with it (Microsoft side has known edge cases on conditional access policies that key on IP). Worth a footnote.

## Policies to refine

- **Policy #1 (Shadow-IT Sanctioning):** Restate the "only sees what hit ZIA" constraint inside the policy box, not just in the Limitations section. Add: AppTotal SSO integration adds OAuth-grant visibility but not log visibility.
- **Policy #2 (Block High-Risk Unsanctioned Apps):** Add policy-precedence note vs URL Filtering and vs Firewall rules. Add: "all File Sharing" category-block is a common Day-1 footgun that catches the corporate sanctioned tenant on first appearance — recommend Tenant Profile allowlist before category block.
- **Policy #3 (Tenant Restriction):** Split Microsoft (TR v2 header injection — requires Entra side configured) from Google (`X-GoogApps-Allowed-Domains`) from Salesforce/Dropbox (org-ID style). State explicitly that Zscaler injects the header; the SaaS enforces. Without that mental model the practitioner blames Zscaler when the policy "doesn't work".
- **Policy #4 (Inline DLP for SaaS Upload):** Add: pinning-driven bypass list. Add: QUIC / HTTP3 handling status. Add: resumable-upload caveat for Google Drive and OneDrive large files. Add: ICAP quarantine destination is rarely deployed in BFSI — note alternatives. Reclassify false-positive risk from "High" to "High, with concrete tuning levers: per-app scope, per-dictionary confidence threshold, dictionary stacking."
- **Policy #5 (SaaS Security API DLP):** Add change-feed latency disclosure per connector (M365 throttling, Salesforce streaming retention). Start-in-Notify is good advice but make it doctrine: "default first 30 days = Notify only" should be stated.
- **Policy #6 (Malware Detection):** State sandbox tier dependency on Zscaler Cloud Sandbox SKU. False-positive risk "Low" is fine; add: post-quarantine restore workflow is a manual admin action, not user-self-service.
- **Policy #7 (External Sharing / Collaborator Governance):** Domain Profile is the single most important object in this whole product — give it its own paragraph. Populate-before-enforce is mentioned; also mention "anyone-with-link" link-rotation behaviour: removing the link in OneDrive does not revoke previously-downloaded copies. That is a practitioner trap.
- **Policy #8 (AI & ML Granular Activity):** Already covered above — add the "what Zscaler actually sees in Copilot" caveat. Add: ChatGPT Enterprise file uploads via the API route are not on the same TLS path as the browser route — Block-Upload via Cloud App Control covers the browser, not the API SDK on an endpoint that talks directly to `api.openai.com`.
- **Policy #9 (CBI):** Reclassify "false-positive risk: Low to medium" to "operational-impact risk: medium to high" — the failure mode is not false positives, it is UX rejection by users. Add cost-per-isolated-session concern.
- **Policy #10 (3rd-Party App Governance):** Add: revocation of an existing grant is not always synchronous with the SaaS — there can be a propagation delay during which the OAuth token still works. Add: "auto-approve" is dangerous in BFSI and should be the first thing turned off.
- **Policy #11 (SSPM):** Drop the compliance-mapping claim until verified. Add: SSPM findings on M365 / Workspace overlap with native posture tools (Microsoft Secure Score, Google Workspace Security Health) — name the overlap so the practitioner does not chase the same finding in two places.

## Limitations the draft missed

- **No inline inspection of QUIC / HTTP3 by default** on legacy ZIA deployments; QUIC is commonly blocked at the firewall to force fallback to HTTP/2-over-TLS so ZIA can see it. Not mentioned.
- **No DLP on encrypted archive uploads (password-protected ZIP, encrypted Office)** in inline mode beyond "block by file type". A user who password-zips a PII file before upload defeats the inline DLP. The API path may still catch it later if the archive is opened in the tenant — but typically not.
- **No native control over browser-extension-driven exfil** (e.g. an extension that POSTs to a developer's webhook). ZIA sees the POST destination as an "uncategorised" host; the user's clipboard / DOM data leaves through a path that looks innocuous.
- **Inspection-bypass list as a CASB blind-spot multiplier.** Every app added to bypass for stability — and there are always some, especially banking-app SDKs and Apple Push Notification Service — is a silent CASB blind spot. The draft does not name this even though it is the single most common operational source of "why did Zscaler not catch this?"
- **Logging / SIEM export latency.** ZIA Nanolog Streaming Service (NSS) feed has its own buffer; investigators looking at SIEM-resident logs for a CASB event may see a 5–15 minute lag vs the ZIA console. Forensics implication.
- **No native DSPM / data-classification-at-rest beyond DLP-rule matching.** Zscaler positions Data Fabric but does not ship a labelling/classification primitive that competes with Microsoft Purview or Google's DLP labels. The draft does not say this.
- **No native integration into Microsoft Purview sensitivity labels for honouring label-based access decisions.** Zscaler can DLP-match content but cannot read a Purview label and act on it the way MDCA can.
- **No native eDiscovery / legal-hold workflow.** The draft says nothing about eDiscovery. Customers expect a CASB to support legal-hold workflows; Zscaler's answer is "use the SaaS vendor's eDiscovery". Worth stating.
- **No DLP for inbound files** in the proxy path (file downloads from a SaaS to the endpoint). Inline DLP is upload-direction-biased. The API path scans at rest. Files actively being downloaded to managed endpoints from an external sanctioned-but-permissive tenant are not the focus. Practitioners assume bidirectional; it is not, in practical terms.
- **No on-premises / air-gapped CASB option.** ZIA is cloud-only. For BFSI customers with on-prem-only deployment mandates (some MAS / BNM segments), this is a deal-breaker that the draft does not name.
- **No native SaaS-to-SaaS data-flow visibility beyond OAuth-grant enumeration.** 3rd-Party App Governance sees the grants; it does not see the actual data flowing through those grants. Tenant-to-tenant data movement (a Box-connected app moving files to a Dropbox account) is invisible.
- **Operational dependency on Zscaler service edges.** A regional ZSE outage takes down inline CASB enforcement. The draft mentions this obliquely at line 313 in the architect-lens note but does not put it in the limitations list. It belongs there.
- **No native consent-revocation rollback / audit-trail export for 3rd-Party App Governance actions.** Once "Revoke grant" fires, reconstructing what was revoked, when, and on whose behalf relies on the SaaS tenant's audit log, not Zscaler's. State this.

## Overall verdict

- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** Snippet-grade sourcing across 30+ "Supported" claims and absent operational footguns (TR v2 mechanics, QUIC handling, pinning-bypass blind spots, Copilot grounding invisibility, change-feed latency reality) mean a BFSI practitioner using this document will draw wrong conclusions; revise per the 12 findings above before promotion.
