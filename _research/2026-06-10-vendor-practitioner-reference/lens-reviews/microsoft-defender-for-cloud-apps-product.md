# Product lens review — Microsoft Defender for Cloud Apps

Reviewer stance: hands-on MDA/MCAS admin since the Adallom days. The draft is unusually honest by vendor-reference standards — it names CAAC's reverse-proxy framing, the 50-file-policy ceiling, the watermark gap, the multi-geo and Power BI delays. That's the floor. The ceiling is higher: there are several quiet footguns and "supported-with-an-asterisk" claims that a practitioner reading this would still walk into.

## Findings to act on (highest impact first)

1. **CAAC session-proxy breakage and UX side-effects are completely absent.** The draft mentions `*.mcas.ms` URL rewriting in passing and never tells the reader that this is the single largest cause of operational incidents in MDA deployments. Apps with hardcoded canonical URLs, anti-CSRF tokens tied to host headers, custom JS that inspects `window.location`, third-party cookies, OIDC iframe re-auth, and SaaS-vendor TOS that forbid URL rewriting (Salesforce historically; some Workday flows) will break or behave unpredictably under CAAC. Microsoft maintains a "Conditional Access App Control known limitations" page; the draft cites neither it nor the `mcas.ms` URL-rewriting caveats. Add a section under deployment-mode for CAAC saying plainly: every CAAC-protected app needs a regression test plan, and some apps will simply not work behind CAAC.

2. **"Block download to unmanaged devices" (Policy #1) device-tag filter is wrong-by-omission.** The draft uses `Device tag ≠ Microsoft Entra hybrid joined AND Device tag ≠ Intune compliant`. That misses **Compliant network** and, more importantly, **certificate-based device identity**. It also leaves the reader thinking these tags are reliable signals — they are not. CAAC sees the device tag the IdP sees at the *session* token-issuance moment; if Conditional Access didn't evaluate device state on that branch (e.g. mobile broker, persistent token), the device tag may be empty and the policy will fail-open or fail-closed depending on `Always apply...if data cannot be scanned`. The draft does not name this. This is the #1 production issue practitioners hit on Policy #1.

3. **"Suspend user (via Microsoft Entra ID)" governance action (Policy #4 and #9) is misleading.** The action does not suspend in the traditional sense — it disables the user object in Entra. Downstream impact: SCIM-deprovisioning waves into every connected app, broken Exchange mailflow if the user is a shared-mailbox delegate, broken Power Automate flows under that account, and the inability for a SOC analyst to *un*-suspend cleanly without resetting credentials. The draft says "auto-suspend" as if it is a soft action. Reword as "disable Entra user object — destructive; downstream SCIM cascades will follow."

4. **DCS (Data Classification Service) latency and quota are not named.** Inline DLP via session-policy "Control file upload/download with inspection" has a real per-file inspection latency budget (typically several seconds for small files, tens of seconds for large) and DCS itself has tenant-level throttling. At BFSI scale during US market open, DCS queueing causes intermittent timeouts that — combined with `always_apply_if_unscannable: false` — fail open silently. The draft writes the YAML flag but doesn't name the operational consequence. This is a regulatory-grade reporting gap.

5. **The "step-up auth (preview)" claim has been in preview for years.** The draft notes it's preview and asks the reviewer to confirm GA. Strong prior: it is still preview. Don't list it as a real capability in a 04-vendors reference; downgrade to "preview, not for production runbooks" and stop mentioning it as an action choice in Policy #3 / #5 mitigations.

6. **OCAS (Office 365 Cloud App Security) deserves explicit Limitations bullet, not just lineage mention.** Practitioners in O365 E5 (not M365 E5) will read this entire document, attempt to build Policy #2/#3/#8, and find none of it possible — no session policies, no third-party connectors, no Generative AI category, no app governance. A single sentence in lineage is not enough; this needs to be in the Limitations section as "if you are on O365 E5 only, ~80% of this document does not apply to you."

7. **Conditional Access App Control requires both an Entra Conditional Access policy AND a Defender for Cloud Apps session policy — and they are configured in separate consoles.** The draft says this once in Policy #1 in passing. It is the single most common cause of "I configured everything in MDA and nothing happened." This belongs at the top of the *Configurable policies* section as a precondition statement, not buried inside Policy #1.

8. **"Suspend user" governance for non-Microsoft IdP tenants (Policy #9) silently degrades.** Draft says "Requires Entra as IdP for the governance actions" — correct but undersold. If the tenant uses Okta/Ping as primary IdP with Entra as resource tenant, the suspend action will disable the Entra shadow object, not the source-of-truth IdP account, and the user will simply re-federate. Practitioner-grade write-up should call this out, not bury it.

9. **Activity policies "auto-disable above 200k/day or 100k/3h" is downstream of a more important footgun.** The Activities engine is per-tenant rate-limited well below those numbers — broad activity policies cause `Match limit reached` alerts that look like normal alerts in the SIEM but actually mean "I stopped matching after N events today, you are blind from this point onward." The draft mentions the auto-disable but not the silent saturation. This matters for compliance attestations of continuous monitoring.

10. **"Per-app manual action: Ban app from OAuth apps page with optional user notification" (Policy #6) understates the kill radius.** Banning an OAuth app is tenant-wide and immediate — every user who consented to that app loses access at the next token refresh. No staged rollout, no per-user revoke without scripting. The draft doesn't warn about this. Practitioners ban a flagged OAuth app at 11pm and wake up to 40 broken integrations.

11. **"Auto-unsanction GenAI apps" (Policy #7) enforcement claim is overstated.** "Tag as Unsanctioned auto-blocks on MDE-managed endpoints" requires (a) the MDE network-protection feature enabled, (b) Network Protection in block mode (not audit), (c) Windows 10 1709+ or specific macOS builds, (d) the endpoint actually reporting to the same tenant. None of those are listed as prerequisites. In real deployments, "tag as Unsanctioned" results in *nothing* happening because Network Protection is in audit mode by default. Name the prerequisites.

12. **No mention of the Defender for Cloud Apps API throttling and the 30-day retention ceiling for activities.** The activities API is hard-rate-limited (documented at 30 requests/min/tenant for the activities endpoint historically; check current values) and activity log retention is 30 days in-product by default — to keep longer you must ship to Sentinel or Log Analytics. The draft has no retention bullet. For a BFSI buyer this is a deal-relevant fact.

## Capability claims to corroborate (do not promote without independent source)

- **"31,000+ app catalogue"** vs **"34,000+"** — the draft itself flags this drift. Pick one with a dated cite, don't quote both.
- **"1,000+ GenAI apps"** — vendor count; no independent corroboration possible. State it as Microsoft's claim, not as fact.
- **"90+ risk indicators"** — vendor marketing number, repeated since ~2019 with no methodology disclosure. Either drop the count or quote it as "Microsoft states 90+ factors across four categories; weighting not disclosed."
- **"Four-times-daily refresh"** of the discovery catalogue — vendor-asserted, no SLA. Practitioner reality is closer to "eventually, when the log collector chooses to flush." Soften to "advertised four-times-daily; in practice delays of 12+ hours are common during high-ingestion windows."
- **"Sandboxing for Box/Dropbox/Google Workspace"** — verify this is still active. Microsoft has historically deprecated file-sandboxing capabilities for non-M365 apps as the Defender XDR detonation pipeline absorbed them. Status as of 2026-06 should be confirmed against the current `anomaly-detection-policy` page, not a 2023 doc.
- **"Catalogue includes Claude, Anthropic API endpoints"** — plausible but exact catalogue contents shift weekly. Don't enumerate specific GenAI vendors in a published reference; say "leading consumer and enterprise GenAI apps including ChatGPT, Gemini, Copilot; full list in the live catalogue."
- **"Defender for Cloud Apps tenant geos: EU/US/UK/AU"** — the draft itself flags this as practitioner-memory. Confirm against the current `disaster-recovery` or `data-security-privacy` doc. AU was added relatively recently; the historical answer was EU/US only.
- **"Automatically enabled with a 7-day learning window"** for UEBA — true but glosses over that the learning window restarts whenever you onboard a new connector, and existing policies will produce false positives during the relearning period. Corroborate the relearning behaviour.
- **"31,000-app catalogue" comparison to other CASBs** — implied parity in the draft; in reality Microsoft's catalogue methodology (signature-based on observed domains) differs substantially from Netskope's CCI or Zscaler's. Apples-to-apples comparison is not possible. Don't imply parity.

## Policies to refine

- **Policy #1 (Block download to unmanaged):** Add the device-tag fail-open/fail-closed warning (finding #2). Add a note that the matching Entra CA policy must include the `Office 365` cloud apps group, *not* individual apps — picking individual apps causes inconsistent CAAC interception across Office workloads.
- **Policy #2 (Auto-label PCI files):** The 50-file-policy hard limit (named in Limitations) bites here first. State that auto-labelling at folder scope often consumes 3-5 policies (per app × per label family) and to plan accordingly. Also: applying a Purview encryption label via MDA does not propagate to existing co-author sessions — users with the file open retain unencrypted access until they reopen.
- **Policy #3 (Block upload of unlabelled PII):** "App filter: all apps marked Enabled on Conditional Access App Control apps page" — there is no such global filter selector; the policy must enumerate apps one at a time. Correct the knob description.
- **Policy #4 (Mass-download anomaly):** "Suspend user" wording (finding #3). Also: 1,000 files in 30 minutes is a noisy threshold for legitimate Teams channel migrations and BI export refreshes; pilot at 5,000 / 30 min, narrow downward.
- **Policy #5 (Block sign-in from non-compliant device):** This is **Access policy**, which is separate from Session policy and has a different evaluation point (sign-in time, not in-session). The draft conflates evaluation semantics. Also Access policies on browser-only break Outlook Web Access for users coming back from PTO who haven't refreshed device compliance — a known annual incident at BFSI customers.
- **Policy #6 (Risky OAuth):** Add the kill-radius warning on Ban (finding #10). Add: `Revoke OAuth app permission` governance action only revokes for the alerting user, not tenant-wide — explicitly contrast with `Ban app`.
- **Policy #7 (Auto-unsanction GenAI):** Add MDE Network Protection prerequisite list (finding #11). Add: the `Generate block script for SWG` action produces a flat IP/domain list that does not match modern GenAI vendor egress (CDN-fronted, JA3-fingerprinted, dynamic domains) and will partially fail at the SWG. Don't sell this as a complete enforcement chain.
- **Policy #8 (Block paste to ChatGPT):** Onboarding ChatGPT to CAAC is non-trivial — requires SAML federation or a non-Microsoft-IdP onboarding flow that OpenAI does not natively support for chat.openai.com (only ChatGPT Enterprise/Teams supports SAML). The draft skips this. State plainly: this policy requires ChatGPT Enterprise/Teams plan, otherwise it is not deployable.
- **Policy #9 (Impossible travel):** "Country-level only, no alerts within the same country or bordering countries" — for Malaysia/Singapore (the user's primary geography) this means impossible-travel between KL and SG produces no alert, which is a material gap for cross-border insider scenarios. Name this explicitly.
- **Policy #10 (Quarantine externally-shared stale files):** `Put file/folder in admin quarantine` for SharePoint moves the file to a hidden admin location; users see a tombstone. Owners are emailed but the email is generic and notoriously ends up in junk. Practitioners run this as `Alert` for 6 weeks before flipping to `Quarantine`.
- **Policy #11 (Malware Box/Dropbox/GWS):** The "Session policy companion" for inline block presumes Box/Dropbox/Google Drive are onboarded to CAAC, which most BFSI shops do not do (CAAC overhead vs limited per-app value). Either drop the companion suggestion or precondition it on prior CAAC onboarding.
- **Policy #12 (Terminated user):** Requires the SaaS account to share a UPN or correlatable identifier with the deleted Entra user. For SaaS apps where the user has a separate local account (the explicit use-case the draft names), correlation only works if the email matches exactly. Common failure: user.surname@corp.com in Entra vs surname.user@corp.com in Salesforce — no match, no alert.

## Limitations the draft missed

- **No native browser isolation / RBI.** Other SSE players (Zscaler, Netskope, Palo Alto Prisma) have native RBI; MDA does not. For risky-category browsing or unmanaged-BYOD, MDA has no answer beyond block.
- **No DLP for email body content in Exchange.** MDA can policy on email *attachments* but cannot inspect message body for DLP — that lives in Microsoft Purview DLP (Exchange policy), a different product surface. Practitioners assume MDA covers email DLP; it doesn't.
- **No CASB-native DRM / EDRM.** Purview MIP labels can carry encryption, but that's Purview, not MDA. State the boundary explicitly.
- **No SaaS posture management (SSPM) for non-Microsoft apps at the depth of Adaptive Shield / AppOmni / Obsidian / Wing.** MDA does *security configuration assessment* for a handful of apps (Salesforce, ServiceNow, GitHub, Okta, Box, Dropbox, ZendDesk) but coverage breadth, control depth, and remediation actions are well below SSPM specialists. Draft positions MDA as "CASB / SSPM" — that framing is generous.
- **No API for policy-as-code management.** Policies are clickops-only in the portal. No Terraform provider, no documented REST API for creating session/access/file policies. Backup/restore of policy configuration is manual JSON export at best. For a regulated environment requiring change control and CI/CD, this is a real gap.
- **No native UEBA risk score for non-Entra identities.** The MDA UEBA score is derived from Entra activity primarily. For users active only in non-Entra IdP'd apps, the risk score is sparse or absent.
- **30-day in-product activity log retention.** To meet BFSI 1-7 year retention requirements, ship to Sentinel/Log Analytics — additional cost.
- **No support for IPv6 in many log-parser sources** for Cloud Discovery. Practitioners with IPv6-heavy egress lose discovery fidelity.
- **No deduplication across data sources.** A user discovered via MDE + Zscaler + Manual upload appears triplicated in the discovery dashboard; user-count metrics are inflated.
- **No real-time DLP on Teams chat content** beyond what Purview DLP for Teams provides. MDA does not policy on chat-message body content.
- **No structured-data DLP** (database-style row/column inspection inside SaaS apps). Salesforce object-level DLP is not a thing in MDA; it is in Salesforce Shield and in some SSPM tools.
- **No customer-managed encryption keys for MDA's own data store.** Tenant data at rest in the MDA service uses Microsoft-managed keys; no BYOK option.
- **No on-prem appliance option.** Pure SaaS — for air-gapped BFSI deployments, MDA is non-viable.
- **No FedRAMP High beyond the GCC High SKU** — for non-US-federal regulated workloads with FedRAMP-High-equivalent demands, the standard MDA tenant is not the right answer.
- **CAAC `Block` action returns a generic error page in some SaaS apps**, not the configured custom block message — depends on the SaaS app honouring the redirect. Don't promise the user-message UX universally.
- **No native integration with non-Microsoft DLP engines.** If the customer has Symantec DLP / Forcepoint DLP / Proofpoint as the enterprise DLP and wants MDA to enforce their existing classifications, the answer is "rebuild your classifications in Purview." That's a multi-quarter project the draft doesn't warn about.

## Overall verdict
- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** Strong skeleton with honest caveats, but missing the most-cited practitioner footguns (CAAC app-compatibility breakage, "suspend = disable Entra object" cascade, DCS latency fail-open, MDE Network Protection prerequisites, no policy-as-code, OCAS-vs-MDA licensing collision) — needs one revision pass to be safe for BFSI consulting use.
