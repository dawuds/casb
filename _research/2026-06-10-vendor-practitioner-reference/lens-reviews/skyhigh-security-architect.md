# Architect lens review — Skyhigh Security (Skyhigh Cloud / CASB)

## Findings to act on (highest impact first)

1. **PKI rollout for SSL inspection is asserted but never engineered** — line 15 says "SSL/TLS inspection is required for most inspection use cases" and lines 53, 109, 133 repeat the dependency, and that is the end of the discussion. In a hybrid M365 + Workspace, Win + macOS managed + iOS/Android BYOD estate this is where day one dies: Skyhigh tenant CA must be pushed to Win system store + macOS keychain + Firefox NSS store + JDK cacerts + Node/Python/Go trust bundles via Intune/Jamf; you need a published bypass list for cert-pinned banking apps, MS Authenticator, Apple Push, MDM, FIDO2 attestation endpoints, and core-banking thick clients **before** the agent ships; and you have **no CA-injection story on unmanaged BYOD**. Draft is silent on every layer. This is the most important deployment fact in the entire document and it is not in the document.

2. **Identity dependency is the whole game and the draft barely sketches it** — reverse proxy (policies #3, the entire "historical differentiator" pitch at lines 16, 214) sits in the SAML flow of the IdP that the SaaS trusts. The brief said hybrid Entra + Okta legacy. The draft never names which IdP the SAML rewrite is authored against, never addresses what happens for apps still federated to Okta, and never addresses how M365 mobile native apps (modern-auth WAM/MSAL, not SAML browser flow) interact with a reverse proxy designed around `login.microsoftonline.com` SAML. The reverse-proxy pitch is **agentless for browsers on federated web apps only**. The draft sells it as "works for BYOD" (line 16) which is misleading.

3. **Skyhigh Client vs. existing endpoint agents is unmentioned** — line 15 names "Skyhigh Client (endpoint agent)" as the forward-proxy path. In a real BFSI fleet the endpoint already carries: EDR (Defender/CrowdStrike/SentinelOne) with network filter, an existing SWG client (Zscaler ZCC or Netskope Client on the legacy estate during migration), a DLP agent (often Forcepoint/Trellix endpoint DLP — sister product, same lineage), MDM/Intune, and possibly a VPN client. Skyhigh Client uses a Windows WFP filter + macOS Network Extension; coexistence with multiple kernel/extension network shims is a known failure mode. Draft does not flag it. The McAfee/Trellix lineage (line 8–10) makes the Trellix Endpoint DLP coexistence question particularly sharp — vendor copy says "unified," operationally these have been two consoles, two agents, two policy authors.

4. **No PoP / latency / TLS-compute budget anywhere** — the brief asked. Draft is silent. Skyhigh runs its SWG/CASB inline plane on its own cloud (formerly McAfee MVISION, now Skyhigh) — PoP locality from KL matters (likely closest is SG, possibly HK; **the draft does not enumerate APAC PoPs at all**). Realistic budget for inline TLS decrypt → DLP classifier → re-encrypt: 20–60 ms added RTT in-region, 2–4× CPU per flow on the egress device when bypass lists are tight. RBI (referenced policy #9 paired with Skyhigh RBI) adds another inline render hop. Practitioner reading the draft underestimates user-perceived latency.

5. **Reverse-proxy positioned as a differentiator without naming the structural erosion** — lines 16, 214 sell reverse-proxy. The draft *itself* admits at lines 176–177 that mobile native apps + cert-pinned apps + non-SAML auth flows bypass reverse proxy entirely. The list of "documented reverse-proxy targets" (line 16: Salesforce, ServiceNow, Workday, Box, etc.) is **exactly the list of apps whose mobile native clients dominate real usage**. This means in 2026 reverse proxy covers a shrinking sliver: managed-browser sessions to federated SaaS. That is honest and needs to be said upfront, not be the historical-differentiator hook.

6. **Forward proxy on TLS 1.3 + ECH (line 178) is buried** — this is named as "a structural limit of the inline model, not a Skyhigh-specific bug" — true, and that is exactly why it belongs in the deployment-architecture discussion, not at limitation #5. The roadmap for ECH adoption (Cloudflare on by default, Chrome rolling, iOS following) means SNI-based steering and inspection degrade over the contract horizon. A 3–5 year SSE bet without an ECH posture is naive.

7. **Where this sits next to SWG / ZTNA / FWaaS / DLP-at-rest is half-stated** — line 4 lists Skyhigh SWG, Private Access (ZTNA), DLP, DSPM and stops. Reality the draft does not address:
   - **CASB Inline + Skyhigh SWG share the same forward proxy and DLP engine** — they are the same data path, separately licensed. "Bought both" is a SKU distinction, not an architecture one.
   - **Skyhigh Private Access (ZTNA)** competes for Skyhigh Client agent attention with EDR/MDM/existing SWG.
   - **Skyhigh DSPM** (mentioned line 4, never explained) is the data-at-rest classifier across cloud storage; it overlaps with sanctioned-DLP-API-mode scanning *and* with Purview/Drive-native DLP that the customer likely already pays for in E5/Workspace Enterprise.
   - **No on-prem file-share DLP, no endpoint DLP, no email DLP** — the draft says this at line 184 ("Endpoint DLP / unmanaged-USB egress... a separate endpoint-DLP tool is required") which is good, but does not say the Trellix Endpoint DLP that customers used to buy alongside McAfee MVISION is now a **separate company's** product post-2022 carve-out (line 10). The "unified DLP" pitch is dead unless explicitly engineered.

8. **"Unified DLP" claim flagged in notes (line 217) is the architect's single biggest contractual question** — sanctioned DLP, web/SWG DLP, and any endpoint DLP are *not* the same policy object in the public docs. The author flagged this for "verify whether one classifier authored once fires consistently across inline, API, and endpoint DLP paths." This is a buying-decision-grade question, not a footnote. Operationally: if you must author the same PII classifier three times, you will get policy drift between inline-block and API-quarantine within a quarter.

9. **License opacity is acknowledged and proceeded past anyway** — line 12, line 211 admit "Skyhigh does not publish a feature/SKU matrix." This makes every "Supported" capability claim conditional on a quote-time SKU. UEBA (line 28), DSPM, RBI, Connected Apps, GenAI controls — all subject to gating that is not in the draft. A BFSI procurement that treats this draft as a capability ledger will discover at quote-time that half the controls assumed are in a higher tier.

10. **Connected Apps OAuth governance has a blind spot the draft only half-discloses** — line 27 caveat ("consumer Google sign-ins or grants to apps from outside the connected IdP/tenant are invisible") and line 175. Real fail mode: an employee uses a corp-issued laptop, signs into a personal Gmail in a second browser profile, OAuth-grants a doc-AI startup access to that personal mailbox which **also happens to carry leaked corporate attachments**. Skyhigh sees nothing. This is not a Skyhigh-specific defect — it is the same gap in every API-CASB — but the draft must name it as a structural CASB limitation, not a Skyhigh caveat.

11. **Third-party-to-third-party SaaS API traffic is named at line 181 and dismissed in one sentence** — Zapier/Workato/Make.com/native SaaS integrations using their own service principal carry sanctioned-tenant data to unsanctioned destinations and never touch user device or corporate IdP. The draft acknowledges this and moves on; a BFSI architect needs explicit guidance that SSPM (Skyhigh DSPM? a separate tool? Adaptive Shield? AppOmni?) is required to cover this, and Skyhigh CASB by itself does not.

12. **Cloud Connector for Shadow-IT log ingest is treated as plug-and-play** — line 18 lists "syslog, file-share, SCP, FTP" and a short list of supported log formats. Line 218 admits "Shadow-IT log-source list is not enumerated in the public docs." Real BFSI feed is: Palo PAN-OS Cortex Data Lake, ZScaler NSS feed, Fortigate FortiAnalyzer, F5 BIG-IP logs, Bluecoat (still alive in some banks), maybe Cisco Umbrella DNS. **Which of these does Skyhigh Cloud Connector parse natively? Which need a SIEM relay (Splunk/Sentinel) in front?** Draft does not say. This is a multi-week scoping question for a non-trivial estate.

13. **"50,000+ services, 1,900+ AI services, 110 risk attributes" is uncritical vendor copy** — repeated lines 23, 35, 213. The author flagged it as "vendor-asserted with no published methodology" at line 213 — and still repeated it as fact in the capability section. Pull or hedge. Macnica (distributor, line 207) is not independent corroboration. This needs Gartner / Forrester / KuppingerCole / customer-survey corroboration to be safe to publish.

14. **No CVE / CISA / NVD search done** — line 219 admits this. Skyhigh inherits the McAfee MVISION codebase, which had documented CVEs in the McAfee enterprise era. A SaaS SSE bet without a vulnerability-history pass is a 3-year contract with eyes shut. This is a hard prerequisite to promote.

15. **GenAI integration coverage gap (line 216) is buried in notes** — ChatGPT Enterprise + M365 Copilot are native; Claude (Anthropic), Gemini Enterprise (Google's own AI in Workspace), Mistral, Cohere, AWS Bedrock front-ends are **only inline/Cloud-Registry coverage** — meaning no post-prompt-history remediation, no API-mode DLP. If the customer is on Workspace + Gemini Enterprise, the deepest AI control surface in the entire stack is not covered by this CASB. This is a tier-1 finding, not a side-note.

16. **"Apply Protection" via Ionic/Seclore DRM is not the same as native tokenisation** — lines 33, 180, 215 all signal this. The 2014-era pure-play CASB pitch of "format-preserving encryption at the gateway" is effectively gone unless you buy and integrate a partner DRM. The draft is honest about it, but procurement copy and competitive comparisons should not treat Skyhigh's "encryption" as native any more.

17. **Console drift across McAfee → Trellix → Skyhigh is acknowledged, then ignored** — lines 39, 212 admit menu paths have moved. **Every menu path in policies #1–12 is therefore a snapshot, not a contract.** Acceptable disclaimer; unacceptable to write the policy authoring guide as if it is stable. A future reader will hit dead links.

## Capability claims to corroborate (do not promote without independent source)

- **"50,000+ services in the Cloud Registry, 1,900+ AI services, 110 risk attributes per service"** (lines 23, 35, 213) — vendor-asserted, no methodology, no third-party validation in the draft.
- **"~40 native API integrations"** (line 17) — vendor page count, "fluid," not corroborated against a current vendor PDF or independent review.
- **Reverse-proxy app list** (line 16: M365, Salesforce, ServiceNow, Slack, Workday, SuccessFactors, Box, Workspace) — verify each is still listed as a supported reverse-proxy target in the current docs, not legacy ones from the MVISION era.
- **"Inline emulation-based sandboxing" for SaaS malware** (line 32) — quoted from product page, no architecture detail (in-line latency, file-size cap, file types supported, sandbox jurisdiction).
- **"Save evidence to S3/Azure, 250 MB cap"** (line 26) — verify this is current, not a 2022 doc. The 250 MB limit is itself a meaningful BFSI constraint and should be in the limitations list.
- **UEBA "machine learning" claims** (line 28) — generic vendor language; no detail on baseline windows, false-positive rates, tunability, or how the ML model is trained per-tenant vs. global.
- **CloudTrust Ratings methodology** (line 24, "evaluated against ~110 risk attributes, 1–9 scale") — closed methodology, no public scoring rubric. Treat as a vendor opinion, not a risk score.
- **M365 Copilot integration depth** (line 35, 113–119) — "surfaces Copilot interactions to DLP" — verify whether this is interaction-level (prompt + response) or only grounding-source-document level. Materially different controls.
- **"~1,900 AI services tracked"** (line 35) — same methodology question as Cloud Registry total.
- **Skyhigh AI bundle as a coherent SKU** (line 35) — vendor announcement language from 2023; verify what is actually shipping in 2026 and what is still "announced."
- **Trellix Endpoint DLP interop / shared classifier story** — not in the draft at all, needs corroboration that classifiers author once and fire on both Trellix endpoint DLP and Skyhigh CASB DLP, or confirm they are now separate.
- **APAC PoP topology and SLAs** — draft is silent. Confirm KL/SG/HK/Sydney coverage and the published SLA on inline latency before any BFSI architecture commitment.

## Policies to refine

- **Policy #1 (Sanctioned-app DLP, API mode)** — name the M365 Graph throttling / Workspace API quotas explicitly. On a 5-year-old SharePoint Online tenant the initial scan is **weeks-to-months** and generates mass historical-PII alerts against stale ACLs. The current false-positive scenario (marketing OneDrive PDFs) understates the day-one volume.
- **Policy #2 (Inline DLP, proxy mode)** — must call out: (a) the PKI rollout dependency from finding #1, (b) the cert-pinned-app bypass list as an authored artefact, (c) ECH degradation from finding #6. The "false-positive risk: Medium" framing is the wrong axis; the real risk is *blast radius if SSL inspection breaks for a sanctioned app at scale*.
- **Policy #3 (Managed-device-required Cloud Access Policy)** — the conceptual JSON (lines 139–168) is fine as illustration but should be flagged as "not literal" (it is, at line 170, but easy to miss). More importantly: the "macOS keychain reset strips device cert" scenario (line 62) is one of *many*. Add: Win profile re-enrol after Autopilot reset, Intune compliance flap, certificate renewal failure window. Also: define the break-glass path when the cert plane is broken **for the whole org**, not just one user.
- **Policy #4 (Connected Apps OAuth governance)** — needs an explicit "what we cannot see" subsection: consumer-account grants, grants to apps outside connected tenants, OAuth grants given via second browser profile on managed device. The "Calendly auto-revoke" FP scenario is fine but trivial compared to the visibility gap.
- **Policy #6 (Anomalous-access-location / UEBA)** — must name the **VPN exit-IP** problem explicitly. On a BFSI estate where remote users hit corp via SSE PoPs (Skyhigh's own POPs! or another vendor's), Skyhigh sees its own PoP as the login location — useless for impossible-travel detection unless the auth-event source includes the originating client IP. Verify whether the M365 / Workspace API actually carries that and whether Skyhigh consumes it.
- **Policy #8 (Shadow-IT block by CloudTrust rating)** — the regional-fintech-rated-7 scenario (line 102) is real. Add: explicit exception-workflow design (who can request, who approves, SLA) before deploying. Otherwise the helpdesk drowns in week one.
- **Policy #9 (Inline GenAI control)** — coverage gap from finding #15 needs to be in this policy. Today this policy effectively means "ChatGPT free, Bard, Bing, Jasper, YouChat are inline-controlled; **Claude, Gemini Enterprise, AWS Bedrock front-ends, Azure OpenAI direct, self-hosted LLM gateways are inline-controlled only at the URL/category level with no model-aware DLP**." Practitioner needs to know.
- **Policy #10 (M365 Copilot DLP via API)** — Purview-label dependency at line 116 is hand-wavy. Either Skyhigh enforces Purview labels (in which case Purview must be deployed first and the customer is paying twice for label-aware DLP) or it doesn't (in which case the Copilot-prompt-content control is shallower than the line suggests). State which.
- **Policy #12 (Tenant restrictions)** — high blast radius is acknowledged (line 134). Add the **Microsoft tenant-restrictions v2 dependency** (Restrict-Access-To-Tenants header + Restrict-Access-Context, requires TLS inspection on `login.microsoftonline.com` which is itself on Microsoft's "do not inspect" guidance for some scenarios). This is a known gotcha and the draft does not name it.

## Limitations the draft missed

- **No PKI / CA injection on unmanaged BYOD** — see finding #1. Stated nowhere.
- **No mention of M365 modern-auth mobile native apps** vs. the reverse-proxy SAML flow — finding #2.
- **No Trellix Endpoint DLP cohabitation story** — post-2022 carve-out, these are now separate companies' products. Finding #3.
- **No APAC PoP enumeration, no latency SLA** — finding #4.
- **No TLS 1.3 / ECH posture beyond a one-line acknowledgement** — finding #6.
- **No published log-source list for Cloud Connector** — finding #12. A BFSI environment cannot scope this without it.
- **No CVE / vulnerability-history pass** — line 219 admits, finding #14.
- **No "what happens when the cert plane is broken for the whole tenant"** — break-glass path missing.
- **No data-residency story for Skyhigh's own control plane** — the draft addresses customer SaaS data residency at line 30 and 183, but never says **where Skyhigh stores incident data, evidence-save artefacts, DLP matches, OAuth-grant inventory, Cloud Registry hits**. BNM RMiT and personal-data legislation (PDPA / forthcoming PDPB amendments) care about this. For a Malaysian BFSI, Skyhigh's control-plane region matters as much as the SaaS region.
- **No saved-evidence size or jurisdiction story** — line 26 says "encrypted copy to S3/Azure, max 250 MB." Whose S3/Azure? Customer-owned bucket or Skyhigh-managed? In which region? Encryption with whose KMS? Material for evidence preservation under BNM RMiT 11.X and supervisory exam-readiness.
- **No mention of OS support matrix or version-floor** for Skyhigh Client — Windows 10 minimum? macOS Ventura/Sonoma/Sequoia? ARM Mac native? Linux desktop? Critical for managed-Mac BFSI estates and for any DevOps Linux endpoint.
- **No mention of Skyhigh API for SOAR / SIEM ingestion** — incident export to Splunk/Sentinel/Chronicle is a baseline requirement and not addressed.
- **No mention of role-based access control / segregation of duties inside the Skyhigh console** — for audit-grade environments, this is a contractual control.
- **No mention of HA / DR posture for the Skyhigh control plane** — single-region failover? Active-active? RPO/RTO commitments?
- **No mention of CDR (content-disarm-and-reconstruction)** beyond "anti-malware + sandbox" — for BFSI, malicious-macro stripping at upload is increasingly expected.

## Overall verdict

- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** Honest about its own gaps in places, but silent on PKI/identity/PoP/cohabitation realities that decide whether this deploys at all in a Malaysian BFSI hybrid M365+Workspace estate; vendor's own marketing numbers are repeated as fact; reverse-proxy "differentiator" is over-pitched given mobile/pinning erosion; needs the same architect rework Netskope and Prisma got before publication.
