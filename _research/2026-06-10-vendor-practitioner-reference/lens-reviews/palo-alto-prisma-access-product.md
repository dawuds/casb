# Product lens review — Palo Alto Prisma Access (SaaS Security / Inline CASB + Aperture API)

Reviewer stance: have run Prisma Access + Aperture/Data Security for two BFSI tenants. The draft is technically literate but soft on caveats and uncritical of vendor doc wording in several places. It also conflates "documented in a marketing page" with "behaves that way in production". Findings below are ranked by what a practitioner would actually trip on.

## Findings to act on (highest impact first)

1. **Three consoles, not two — and the draft hides one.** The draft repeatedly says "two consoles" / "inline vs. API" (lines 38, 192, 224). In real life as of 2025–2026 you live across **three** UIs: Strata Cloud Manager (SCM) for inline / SaaS Security Inline / AI Access; the legacy **SaaS Security Cloud Management Console (CMC)** for Data Security + SSPM + Behavior Threats (PAN is migrating these into SCM but the migration is incomplete and tenant-dependent through 2026); and **Common Services / hub** for licensing, tenant onboarding, Enterprise DLP profile management. Enterprise DLP itself is configured in a fourth place (Strata Cloud Manager → Enterprise DLP) before being referenced. Practitioners who read this draft and budget for "two policy planes" will be wrong on day one. Rewrite §Lineage and the Limitations bullet on line 192.

2. **"Enterprise DLP profiles are shared between the two" is overstated (line 39).** Yes, the DLP profile object is shared in principle. In practice (a) the API-side Data Security still maintains its own legacy "data patterns/profiles" lineage that PAN has been migrating to Enterprise DLP unevenly; (b) profile parity is asymmetric — some advanced Enterprise DLP features (EDM exact data match, OCR on images, ML-based classifiers) work inline but produce different match behaviour or are unsupported at-rest; (c) profile changes do not version-sync — editing a shared profile silently changes behaviour on both planes with no diff/preview. The draft presents this as a clean consolidation point. It is not. Caveat needed.

3. **"Risk score is computed from 55+ of 80+ tracked attributes" (line 44) is a vendor marketing artefact, not an engineering claim.** The number changes by quarter, by app, and by what data PAN can actually source for that app. Strip the precision. Either source it from a dated doc page with the version number, or rewrite as "vendor-computed score over a documented attribute set; attributes per app vary".

4. **The 25 MB ceiling is incorrectly stated as universal (lines 50, 62, 139, 194, 228).** PAN's per-app matrix documents **different ceilings per app and per scan type** — historically 20 MB on some Google Workspace flows, 50 MB on a couple of newer connectors, and different limits for malware (WildFire) vs. DLP inspection vs. metadata-only. "25 MB across all apps" is the average a marketing slide quotes; the per-app matrix disagrees. The note at line 228 says "I treated this as a hard product limit" — that admission belongs in the draft, not the appendix. Either pull the live per-app matrix or rewrite as "file-size ceilings vary per app and per scan type; commonly 20–50 MB; verify against the per-app supported-applications matrix at deployment".

5. **"Forward-scan only" list is wrong and incomplete (lines 50, 195, 227).** The draft lists Gmail / Google Chat / Zoom / Zendesk / Airtable / ChatGPT as forward-scan-only. Reality: (a) **Slack** historical-message backscan has been forward-only for non-Enterprise-Grid tenants for years — missing from the list; (b) **Teams chat content** scanning is forward-only and depends on Microsoft's Teams Export API throttles that PAN does not control — missing; (c) **Workday** has no content-DLP at all in practice, only metadata/posture — missing from the list and from limitations; (d) **Google Chat** historical scan has shifted in 2024–2025 with Google's Workspace Events API maturity — the blanket "forward-only" is stale. The draft's own footnote (line 227) flags this uncertainty; the body should reflect it.

6. **Reverse-proxy / MDA-equivalent claim (line 34, 202) is correct in spirit but misses a footgun.** Clientless VPN is named as the partial substitute. The draft does not mention that **Clientless VPN is a fundamentally different architecture** (PAN's portal terminates the session and re-renders the SaaS app on the user's behalf — closer to RDS/VDI than to MDA's IdP-routed reverse proxy), with substantial app-compatibility caveats: complex JS (Tableau, certain Salesforce Lightning components, Power BI embeds) breaks; downloads/uploads can be brokered but with file-size and MIME caveats; modern SPA frameworks routinely fail or render with subtle data-handling bugs. A practitioner reading the draft would think "ok, I have a fallback for BYOD". They do not. Limitation needs to be sharper.

7. **"Behavior Threats" detection lift from vendor doc — categories listed (line 54) but real-world FP/FN posture omitted.** Three things practitioners hit and the draft does not warn about: (i) **90-day baseline cold-start** — the dynamic policies are useless for the first ~30 days post-onboarding and noisy for the next ~30; (ii) **Per-user, not per-role** baselining — joiners/movers/leavers generate weeks of FPs after any role change; (iii) **No cross-app correlation** — anomalous-location detection on Salesforce does not know the same user just appeared on M365 from the same IP — PAN does not market a cross-app session graph the way Netskope and MDA do. Add these.

8. **"Compromised-account detection — Supported" with detection-only caveat (line 56) is technically correct but understated.** The footnote at line 226 hedges this; the body should say plainly: **no native session-kill, no native step-up, no native revoke-token**. Response is XSOAR-or-IdP, period. The draft buries this. Promote to a top-line caveat on the capability claim itself, not just the limitations bullet at line 200.

9. **AI Access Security tenant-restriction headers for Claude and Gemini (line 146) — flagged as "verify per app" but the policy is still in the policy table.** This is the dangerous shape: a policy a practitioner can read and copy without realising the underlying capability for two of the three apps is not vendor-confirmed. Either (a) remove Claude + Gemini from Policy #9 and leave only ChatGPT (which does have documented Workspace-ID enforcement), or (b) caveat inline that for non-ChatGPT GenAI apps the enforcement is via app-category / URL-rule + HTTP-header insertion only if PAN's dictionary supports that app's tenant header, which not all do.

10. **HIP "Corporate-Managed" gating (Policy #3, line 95–100) — missing the cert-pinning + non-Windows footguns.** HIP works well on Windows. On macOS (especially post-Ventura SIP/TCC changes) HIP collection has had recurring fidelity gaps; on iOS/Android the HIP report is a thin subset of the desktop one; on Linux the HIP agent is feature-trailing. A "block download from unmanaged" rule built off HIP can let macOS/iOS pass when it shouldn't, or fail safe-by-blocking valid mobile users. The draft makes HIP sound platform-neutral. It is not.

11. **OAuth governance (line 52, 196) — consumer-account caveat correct, but missing the bigger gap.** SSPM third-party plugin discovery is **dependent on the marketplace tenant's admin-API granting the read scope**. For Google Workspace this requires a domain-wide-delegation service account with sensitive scopes — many BFSI security teams cannot get this approved by their Workspace admin, in which case SSPM falls back to a degraded view. The draft assumes the connector "just works". It often doesn't, and the failure mode is silent partial visibility, not an error. Add caveat.

12. **"Encryption / tokenisation — Not supported" (line 64) — accurate but the reason given is wrong.** "Absence from product docs" is not the right citation. The right framing: PAN never built the proxy-injected encryption pillar that early CASB vendors (CipherCloud, Skyhigh-pre-McAfee) marketed because the SaaS vendors broke the integrations and most enterprises walked away from it. State this as a design choice, not a documentation gap.

13. **Sanctioned-app list of ~28 apps (line 35) is presented as a fact but is a snapshot.** The list mixes apps with full content-DLP, malware, and remediation actions with apps that have only metadata/posture/forward-scan. As written it implies parity across all 28. There is no parity. The supported-apps matrix has columns for: at-rest DLP, malware, share-link control, user activity, quarantine — many "supported" apps tick only 2–3 of these. Rewrite as "~28 sanctioned-app connectors with **highly variable feature parity**; consult the per-app matrix".

14. **Strata Logging Service ingestion lock-in (line 36, 193) — correct, but practitioner consequence understated.** The draft says no third-party-firewall log ingestion. The practitioner consequence: in any BFSI shop with **on-prem inspection that is not PAN NGFW** (Check Point, Fortinet, Cisco Firepower), the shadow-IT view is **missing the entire on-prem egress fraction** of the user population. For a partially-cloud workforce this is a 30–70% blind spot, not a niche edge case. Sharpen.

15. **No mention of Strata Cloud Manager UI latency, search, and audit-trail footguns.** SCM is comparatively new and has known practitioner issues: log search has timeouts on broad queries, the policy commit pipeline is opaque (you push a recommendation, it shows up as a security rule but the lineage view is partial), and admin audit logs for SaaS Security policy changes are split between SCM audit and CMC audit. Anyone managing this in production needs to know.

## Capability claims to corroborate (do not promote without independent source)

- **"55+ of 80+ tracked attributes" for risk scoring (line 44)** — vendor doc page only; no independent corroboration; numbers drift.
- **"Behavior Threats uses up to 90 days of historical data" (line 54)** — vendor doc only; in practice the baseline window appears to be 30/60/90 selectable per policy type on some tenants; verify in CMC.
- **"AI Access Security — GA Jan 2024" (line 10)** — vendor blog/marketing; pin the GA date from a PAN release-notes URL, not the SASE landing page.
- **Tenant-restriction header support for Claude and Gemini via AI Access (line 146)** — not corroborated by Anthropic or Google docs in the draft. Pull the Anthropic Workspaces docs (Claude) and the Google Workspace tenant restriction v2 docs (Gemini) and confirm what header schema each accepts. If Claude/Gemini don't publish a header-based tenant restriction, the policy as written is fiction.
- **Microsoft Learn cross-link (line 108)** — this page is about **Microsoft Entra Global Secure Access coexistence with Prisma Access**, i.e. when both products are deployed. It does corroborate that PAN can insert tenant-restriction headers, but does not validate PAN's AI Access GenAI tenant-restriction claim. Don't let this single non-PAN source carry weight beyond what it actually says.
- **"~90 apps for SSPM posture" (line 201)** — vendor figure; check the SSPM supported-applications page for the exact count by quarter; this number moved meaningfully in 2025.
- **"WildFire-backed scanning" of files at rest (line 62)** — verify whether WildFire verdict is fetched on every scan or cached from inline inspection; this matters for the latency-to-quarantine SLA practitioners need to quote.
- **Strata Logging Service rename from Cortex Data Lake (line 36)** — correct, but PAN's docs are inconsistent across pages; some 2025 doc pages still use the old name. Note version.

## Policies to refine

- **Policy #1 (Block upload to unsanctioned GenAI):** The draft says "Policy Recommendations (pushed as a Security Policy rule)". The Policy Recommendations engine generates *suggestions* — the security policy rule still has to be committed by an admin into the Prisma Access security policy, and the rule order matters (it gets inserted at a position the recommendation engine picks, which is rarely where you want it for a deny rule). Add the rule-ordering caveat. Also: "Block + custom block page" requires decryption *and* the custom-response-page feature being enabled on the firewall profile — note this.

- **Policy #2 (Quarantine externally-shared Drive PII):** Good shape. Two missing knobs: (a) **Shared Drive ownership semantics** — files in a Shared Drive don't have an "owner" in the My Drive sense; the quarantine remediation behaves differently (admin folder copy + tombstone vs. permission-strip). The conceptual YAML conflates the two. (b) **Quarantine "tombstone" leaves a placeholder** — end users see a file disappear and an admin-folder pointer they can't access; the helpdesk impact needs a comms plan. Add as practitioner note.

- **Policy #3 (Block download to unmanaged):** See finding #10 — add HIP platform-fidelity caveat. Also: HIP-match in the security policy rule is evaluated **at session establishment**, not continuously — a device that passes HIP at session start and then has its compliance state change mid-session keeps the session. State this.

- **Policy #4 (M365 tenant restriction):** The Microsoft tenant restriction model has **v1 and v2**, with different header schemas and enforcement behaviours. The draft does not distinguish. v2 is what most BFSI tenants want (it covers Microsoft auth and downstream graph calls); v1 is older and lighter. Specify which the policy targets and confirm Prisma Access supports the v2 headers — Microsoft Learn page at line 108 is the right place to pin this.

- **Policy #5 (Impossible travel Salesforce):** "Vendor default ~kmh" is hand-waving. The actual default is documented (~800 km/h commonly); pull it. Also: Salesforce-side audit log freshness governs how quickly this policy can fire — PAN polls Salesforce, latency from event to alert can be 5–30 minutes, which is too slow to qualify as a real-time control. State the detection-to-alert SLA honestly.

- **Policy #6 (Revoke OAuth):** Auto-revoke for "high vendor-risk score" without a published rubric for how that score is computed = audit risk. Practitioners need to know the score is **vendor-opaque** and changes over time, which means a previously-allowed app can be auto-revoked next quarter without any policy change. Add caveat. Also: revoke on Google Workspace requires the SSPM service account to hold a sensitive scope — see finding #11.

- **Policy #7 (Bulk download Box outside hours):** Two issues. (i) "User-local" time requires either user-attribute mapping (IdP-sourced time zone) or PAN's geo-IP best-guess — clarify which. (ii) Box's audit log granularity for downloads has historically had **per-event vs. aggregated-event** quirks that affect what "n downloads in an hour" actually counts. Add caveat.

- **Policy #8 (WildFire on OneDrive):** Add the file-size ceiling concretely (see finding #4) and the **WildFire verdict latency** — verdicts can take minutes to hours for unknown samples, so the "quarantine on malicious verdict" remediation may fire long after the file was shared further. This is the policy where latency matters most.

- **Policy #9 (GenAI tenant restriction):** See finding #9. As written, this policy is dangerous to copy.

- **Policy #10 (Coach away from Tolerated):** "Once per user per day" — confirm this is a configurable frequency on the inline response page or whether the practitioner is approximating from the security rule's session-lookup. In my experience the frequency control is per-session, not per-day, and "per day" requires custom logic. Verify.

## Limitations the draft missed

- **No CASB-side malware sandboxing alternative to WildFire.** If your Group Risk team has rejected sending files to WildFire's cloud sandbox (data-residency, regulator-sensitive content), you have **no in-product alternative**. The malware scanning capability collapses. The draft does not name this.
- **Enterprise DLP OCR is async and has size + format caveats.** Image-borne sensitive data (screenshots of card numbers, IDs photographed in chat) is matched in inline DLP via OCR only if the OCR engine catches it; the matrix of what OCR runs on what content type at what size is non-trivial and undocumented in customer-readable form.
- **Cross-tenant collaboration in M365 / Workspace is poorly handled.** API-side scanning sees the file in *your* tenant; if a corporate user collaborates into an external tenant, the scope ends at the tenant boundary. Inline can catch the activity, but the API plane goes blind. Worth naming.
- **No native CASB-as-DSPM.** No data-classification map of "where in our SaaS estate does PII live" without building it yourself from incident outputs. PAN's DSPM positioning lives in a different product (Prisma Cloud DSPM), not in SaaS Security.
- **SCM-CMC migration risk.** The platform is mid-migration. Features documented in CMC may move into SCM with subtle behaviour changes mid-contract. Procurement teams should know.
- **No native session-recording / replay for high-risk users.** Some compliance regimes expect this; PAN does not offer it on the CASB plane.
- **Admin RBAC granularity is coarse.** Separating "can view Data Security incidents" from "can run remediation actions" from "can edit policies" is doable but requires Common Services role configuration that is its own multi-step setup; this gets missed in BFSI four-eyes-control deployments.
- **No native data-subject-access / right-to-erasure helper.** When a DSAR comes in and you need to find a person's content across sanctioned SaaS, PAN's API CASB will not help you — its index is incident-shaped, not subject-shaped.
- **Limited Mac/Linux GlobalProtect parity** — see finding #10. The draft frames GlobalProtect as a platform-agnostic agent. It is not, in practice.
- **No native control over GenAI prompt content leakage via the GenAI app's *output* side.** AI Access inspects prompts (uploads). Data exfiltrated through a model's *response* (e.g. the model regurgitating training data) is not a CASB concern PAN can address. Worth saying so explicitly because customers ask.
- **Behavior Threats and Data Security incidents are not unified into a single case-management view.** Incident triage spans two surfaces. The XSOAR integration helps but is paid and separate.
- **Per-tenant rate limits on SaaS app APIs are PAN's hidden ceiling.** When Microsoft / Google / Salesforce throttle PAN's API calls, scan latency and remediation latency degrade silently. No customer-visible metric for "are we API-throttled right now". Practitioners discover this when remediations stop firing.

## Overall verdict

- **Promote as-is to 04-vendors/?** NO
- **One-line reason:** The draft is technically literate but lifts vendor doc phrasing without enough caveats on the three-console reality, the per-app feature parity gap, the 25 MB ceiling, the response-action ceiling (no native session-kill), and the BYOD/reverse-proxy gap — promote after rewrites to findings 1–15.
