# System integration / platform engineer - playbook v0 review

Reviewer stance: system integration / platform engineer. Concern: wire mechanics between MDA and the rest of the stack — SIEM, SOAR, ticketing, IdP federation, downstream SCIM provisioners, Graph API rate budget, multi-geo tenancy, DR / BCP. The architect lens looked at deployment topology. The product lens looked at console workflows. This lens looks at **what flows where, over which connector, at what throughput, and what breaks when it doesn't**. Vendor draft and three prior lens reviews read; findings below go *beyond* what they raised.

## Highest-impact findings (rank-ordered)

1. **Section "What this playbook does NOT cover" / line 230 — "Build SIEM queries that correlate `CloudAppEvents` with Entra Sign-in logs by session correlation ID."** This is the only SIEM guidance in the entire document and it is *wrong by omission*. There is no documented "session correlation ID" common to `CloudAppEvents` and `SigninLogs`. The practitioner-grade join is **`AccountObjectId` + `IPAddress` + `TimeGenerated` ±30s window**, plus `ApplicationId` where present. `CloudAppEvents` carries `AccountObjectId` (Entra ObjectId GUID) and `IPAddressLocation`; `SigninLogs` carries `UserId` (same Entra ObjectId) and `IPAddress`. There is no shared session-correlation field. **A practitioner who writes the join the playbook tells them to write produces an empty result set.** Fix before promotion.

2. **Lines 49 / 230 — "Forward to Sentinel for retention beyond 30 days" with no connector specification.** There are *three* paths and they are not equivalent: (a) **Microsoft Defender XDR data connector** in Sentinel — single connector, ingests `CloudAppEvents` + `AlertInfo` + `AlertEvidence` + all other Defender XDR tables, billed at Sentinel ingest rate (~USD 2.30/GB analytics-tier 2026 list, varies by commitment tier and region), **deduplicated against the Defender XDR free 30-day Advanced Hunting pool for the first 30 days only on those specific tables documented in the connector — read the current docs, Microsoft has changed this twice**; (b) **Office 365 connector** — covers OfficeActivity not MDA; not a substitute; (c) **Diagnostic Settings on Entra ID** — only ships `SigninLogs` / `AuditLogs`, *not* MDA events. The playbook conflates these. Day 90 SOC team will pick the wrong one and either double-bill or get incomplete data.

3. **All Day 30 / Day 90 policies omit the "what gets emitted to the SOC stack" mechanics.** No mention of: (a) Microsoft Graph Security API as the canonical export surface for alerts to a third-party SIEM (Splunk / QRadar / Chronicle) when Sentinel is *not* the destination; (b) the deprecated **SIEM Agent for MCAS** (CEF-over-Syslog from a Java agent the customer hosts) — Microsoft has signalled deprecation; the playbook should explicitly say "do not deploy SIEM Agent for new builds, use Graph Security API or Sentinel"; (c) the Microsoft Sentinel connector for **Defender XDR** ingests alerts but **does not** ingest the full activity stream by default — you opt in tables individually and `CloudAppEvents` is one of them, with separate billing. For a Splunk-anchored bank without Sentinel, the integration path is undocumented in the playbook.

4. **Graph API throttling on Policy 2 (auto-label PCI) and Policy 7 (quarantine externally-shared) — not named anywhere.** File policies in MDA pull file content / metadata through Microsoft Graph and SharePoint REST under the **tenant's app-only Graph budget**. Throttling thresholds (current Graph throttling guidance): SharePoint Online ~600 requests per app per minute per tenant for delegated, with backoff via `Retry-After` headers; OneDrive search and file-read calls hit separate buckets. **At BFSI scale (50k+ users, multi-TB OneDrive), tenant-wide file policy on day 1 saturates the Graph budget within hours, and MDA backs off to "partial coverage" — it does not queue indefinitely, it skips and re-evaluates on next pass with no operator notification of the gap.** Policy 2 / Policy 7 must be scoped (parent-folder-narrow) on rollout for Graph-budget reasons, not just DCS-quota reasons (which lens-product raised). Add this.

5. **"Suspend user" SCIM cascade (Policy 3, Policy 10) — direction and latency unspecified.** "Suspend user" disables the Entra user object (correctly named in the playbook). What the playbook does *not* say: Entra is typically the **SCIM source** for downstream apps (Salesforce, ServiceNow, Workday, Slack, Zoom, GitHub Enterprise, Box, Dropbox, Atlassian Cloud). Disable in Entra → SCIM `PATCH /Users/{id}` with `active: false` → downstream app deactivates. **Latencies (Entra SCIM provisioning cycle):** default 40-minute incremental sync; on-demand provisioning available per-user; full-sync only on initial / reset. Apps that **will see the disable event automatically**: any SaaS configured under Entra Enterprise Applications → Provisioning. Apps that will **NOT** see it: (a) apps provisioned by Okta / OneLogin / JumpCloud (not Entra), (b) apps with their own user store not federated for provisioning (most legacy on-prem, many smaller SaaS), (c) AWS IAM Identity Center if not Entra-federated, (d) local accounts in any SaaS bypassing SSO. **The playbook tells the practitioner that "Suspend user" cascades destructively but never tells them which way the SCIM arrows actually point in their estate.** This is the single biggest operational unknown at deployment time.

6. **Okta-fronted SaaS + CAAC (Day 30 / Day 90 policies) — playbook is silent on the actual onboarding mechanics.** Architect lens raised the cost; this lens raises the *mechanics*. To CAAC-protect an Okta-fronted app: (a) the app must federate to Entra (either as a secondary IdP via SAML, or by re-pointing the SaaS SSO from Okta-as-IdP to Entra-as-IdP and Okta-as-Federation-Hub via WS-Fed/OIDC chain), (b) Entra CA policy with "Use Conditional Access App Control" must trigger on that app, (c) the MDA session policy must reference the Entra app object. **What breaks:** Okta Verify device trust (the Okta session cookie no longer governs the post-Entra hop), Okta FastPass (cert-bound to Okta-as-RP — Entra is now RP), Okta Adaptive MFA inside the SaaS session (Okta no longer mediates the in-session signal), Okta Workflows triggered by Okta sign-in events (Entra sign-in does not trigger Okta hooks unless explicitly wired). **Net: every Okta-fronted CAAC onboarding is a 2-4 week project per app, not a checkbox.** Add this to the Day 30 / Day 90 effort budget.

7. **CAAC + Cross-Tenant Access Settings (XTAS) + B2B guest sessions — completely absent.** A BFSI tenant federated to vendor tenants (audit firms, consultants, fund managers) for joint operations has B2B guest users in the home tenant. **CAAC behaviour on B2B guests is non-trivial**: the home-tenant CA policy can target B2B guests (`Users and groups → Guest or external users`), but the device-tag claims (`MicrosoftEntraHybridJoined`, `IntuneCompliant`) come from the **guest's home tenant**, not the host tenant. If the guest's home tenant doesn't surface the claim, the policy fails-open or fails-closed depending on `Always apply if data cannot be scanned`. The playbook never mentions XTAS or B2B guests. For Policy 5 (block download on unmanaged) this is decisive — auditors signed in as guests will either get blocked entirely (operational pain) or fail through with no block (control failure).

8. **M365 multi-geo / multi-region SharePoint — silent.** Compliance lens raised the privacy angle; this lens raises the *integration* angle. In a multi-geo tenant (e.g. primary MY, satellite SG, satellite AU), each SPO geo has its own Graph endpoint and its own activity stream. MDA Office 365 connector pulls from the primary geo's audit pipeline; **satellite geos surface in the unified audit log with `OrganizationId` + `GeoLocation` fields, but MDA's file-policy scan path may or may not iterate them — Microsoft documents this poorly**. The playbook says "Power BI / Dynamics 365 24–72hr latency" but never says "satellite SharePoint geos: verify per-geo coverage." Practitioner walking blind into multi-geo will get incomplete file-policy coverage and an audit gap they don't notice.

9. **DR / business continuity — entirely absent.** If `*.mcas.ms` proxy region is unreachable: (a) CAAC session policies fail to a **per-policy fail mode** that is *not* a global toggle — each session policy has its own `Always apply` behavior, and the playbook's hardcoded `false` (line 120) means **on proxy outage, downloads are allowed through** (fail-open). For a BFSI control on PCI/PII data, that is a control-failure event. (b) API-mode policies queue at MDA's backend; on regional MDA outage, file-policy matches stall but historical events are not lost (Microsoft claims). (c) Governance actions in flight: a "Suspend user" or "Quarantine file" issued before outage may not execute; no operator notification. (d) Alerts queued for SIEM forwarding: ingestion lag rather than data loss for Sentinel; CEF/Syslog feeds may drop. **None of this is documented. Add a BCP section before promotion.**

10. **Power Automate as a governance-action hook — not mentioned at all.** MDA has a "Send to Power Automate" governance action that wires a policy match into a Power Automate flow (for ticketing, Teams notification, custom remediation). Known issues a practitioner *will* hit: (a) Power Automate runs under a service-account identity that requires its own licensing (Power Automate Premium for HTTP connectors to ServiceNow / Jira), (b) Power Automate has its own throttling (~6000 actions/5min for the default action limit), (c) outbound HTTP connectors from Power Automate run from an Azure datacentre IP pool that **changes** — corporate firewall allowlisting via static IP breaks weekly; use service tag `AzureCloud.{region}` instead, (d) flows under a personal account get orphaned if that account is offboarded — a textbook MDA + Power Automate footgun. The playbook should explicitly mention "do not author governance-action flows under personal accounts; use a service principal with documented owner."

11. **Ticketing-system integration (ServiceNow / Jira) for Day 1 SOC workflow — undocumented.** The playbook's Day 1 policies emit alerts; alerts need tickets. Paths: (a) Sentinel → ServiceNow ITSM connector (uses ServiceNow's Event Management or Security Incident Response — different table, different schema); (b) Defender XDR → ServiceNow via the **Microsoft Sentinel SOAR / Logic Apps** native connector (alert in, incident out); (c) Power Automate → ServiceNow custom connector; (d) email-to-ticket (worst option, but commonly used). The playbook says "Alert + email to SOC" for Policies 3 and 4 — this is fine for Day 1 but **does not scale**. Add a "from Day 30 the SOC must have a connector path, not email-parse" instruction.

12. **The "two consoles" warning (line 50, Policy 5 / line 119) is correct but lifecycle drift is unnamed.** The playbook says Entra CA + MDA session policy are both required. What it does NOT say: (a) **policy drift** — Entra CA policy can be edited or expired by the Entra admin without the MDA admin knowing; the MDA session policy becomes an *orphan* that never fires; auditors will find a configured-but-inert MDA control with no alerts. (b) **policy ordering** — Entra evaluates CA policies in parallel and the "session control" outcome is determined by policy intersection; if a separate CA policy also targets the same scope with "Sign-in Frequency = 1 hour", the user re-auths through a path that may or may not hit the CAAC session control depending on token cache. (c) **break-glass** — neither console publishes a "policy stack export" that gives the auditor a single view of "what CA + MDA policies govern this app for this user." Cross-tooling export is manual JSON via Graph + manual MDA portal export. Name this; it's the #1 audit finding I'd write up.

## Specific fixes the v1 must apply

- **[Section "What to do next" / item 8]** Rewrite the SIEM-query guidance. Replace "correlate by session correlation ID" with: "Join `CloudAppEvents` and `SigninLogs` on `AccountObjectId` = `UserId` within a ±30s window on `TimeGenerated`, intersected on `IPAddress`. The audit trail is **three** records correlated: Entra `SigninLogs` (auth event), Entra `AuditLogs` (CA policy result), Defender XDR `CloudAppEvents` (session-policy match). All three required for evidence."
- **[Section "What it touches"]** For "Microsoft Sentinel / Log Analytics" row, add the connector list: **Defender XDR data connector** (canonical for MDA tables — `CloudAppEvents`, `AlertInfo`, `AlertEvidence`); not Office 365 connector, not Diagnostic Settings on Entra alone. State the dual-bill risk: Sentinel ingest on top of Defender XDR retention.
- **[All Day 1 / Day 30 / Day 90 policies]** Add a row to each policy table: **"SIEM artefact and connector"** — specifying the Defender XDR table, the Sentinel connector that carries it, and (for non-Sentinel customers) the Graph Security API endpoint for the alert pull. Without this, the playbook is "configure this" with no "verify this fired" path.
- **[Policy 2 / Policy 7]** Add a "Graph API rate budget" line — file policies pull through Graph; at tenant-wide scope they saturate; scope by parent folder on rollout; monitor via Microsoft Graph rate-limit telemetry.
- **[Policy 3 / Policy 10 — "Suspend user" trap blocks]** Add the **SCIM cascade table**: which downstream apps in a typical BFSI estate will see the disable event automatically (Entra-provisioned apps via Enterprise Applications), which need manual intervention (Okta-provisioned, local accounts, non-federated AWS IAM, B2B-guest sessions in vendor tenants). Latency = 40-min default; on-demand provisioning available for incident response.
- **[Day 30 / Day 90 / CAAC sections]** Add a **per-app CAAC onboarding effort estimate**: standard M365 / Salesforce / Box / Workday on Entra IdP = 1-3 days including regression test; Okta-fronted SaaS requiring re-federation = 2-4 weeks; cert-pinned native app = unsupported, drop from scope; B2B-guest CAAC = case-by-case, expect to fail-open or fail-closed per device-claim availability. The playbook tells practitioners to "budget for per-app regression testing" but does not size it.
- **[New section before "What to do next"]** Add **"Disaster recovery / continuity"**: per-policy fail mode for CAAC outage (each session policy's `Always apply if data cannot be scanned` is the per-policy fail mode — current default is fail-open); API-mode governance-action behaviour on regional MDA outage (queued vs lost); SIEM forwarding behaviour on Sentinel ingest stall.
- **[New section]** Add **"Multi-tenant / multi-geo behaviour"** — M365 multi-geo SharePoint coverage (verify per-geo iteration before claiming tenant-wide compliance evidence); Cross-Tenant Access Settings interaction with CAAC (B2B guests get device claims from home tenant — failure mode); satellite-tenant alert routing (Defender XDR alerts surface in the security tenant the user authenticates against).
- **[Section "What this playbook does NOT cover"]** Replace the existing "two consoles" line with a **policy-drift detection** recommendation: schedule a Graph API query against `policies/conditionalAccessPolicies` and an MDA portal export comparison monthly; an orphan MDA session policy (no matching CA policy) is a high-severity finding.
- **[Policy 8 — auto-unsanction GenAI]** Add the Power Automate / SWG export caveat that already exists, AND add a SCIM-cascade-for-blocked-app note: when MDA tags ChatGPT as Unsanctioned and MDE Network Protection blocks the egress, **users with pre-existing OpenAI personal accounts still authenticate via personal IdPs out-of-band** — the integration "block" does not cover this. The playbook says "consumer ChatGPT bypasses" elsewhere but not here.
- **[Policy 9 — clipboard paste block on ChatGPT]** State that DLP-content inspection on clipboard creates a high-volume telemetry stream into `CloudAppEvents` that **inflates Sentinel ingest cost** measurably. Real-world: clipboard inspection on 5k seats at typical paste frequency = ~100GB/mo additional Sentinel ingest. Budget impact, not just privacy impact.

## Claims to verify or remove

- **Line 230 "session correlation ID"** — does not exist in the documented schema. Remove or rename to `AccountObjectId` + IP + timestamp window.
- **Line 70 "subscribed weekly discovery report"** — verify the report retention policy and whether the report is signed / tamper-evident. If it's a CSV export with no integrity claim, it's not an audit artefact.
- **Line 105 "ML-based, ignores VPNs and tenant-common locations (Microsoft's claim)"** — already flagged in lens-product; this lens adds: the VPN-detection signal feed is a Microsoft-internal IP intelligence database; on outage the policy reverts to country-only mode silently. Verify whether the policy *signals* the SOC when its enrichment source is unavailable. I doubt it does.
- **Line 50 "Two consoles. Two policy objects. Both required."** — correct but understated. The third object is the **Entra app registration** (the enterprise app that mediates the SAML flow for the SaaS being CAAC'd). For non-M365 apps this is a third configuration object that can drift independently.
- **Line 165 "App-tag-change event in `CloudAppEvents`"** — verify the event class name. `CloudAppEvents.ActionType` has changed over years; my prior is `"Update app tag"` or `"App marked as unsanctioned"` but Microsoft has renamed. Confirm.
- **Line 188 "Maps to NIST CSF PR.AA-05 (access permissions)"** — fine as illustrative, but the playbook is mixing illustrative compliance mappings into operational policy tables. Either pull all such mappings into a dedicated section or strip them; mid-table illustrative cites are a precision-rot risk.
- **Lines 49 / 230 "ship to Sentinel"** — verify the actual cost. As of 2026, Sentinel ingest is tiered and commitment-discounted; the 30-day free pool overlap with Defender XDR is documented but the rules have changed. Pull from the **current** Sentinel pricing page and Microsoft Defender XDR-Sentinel integration doc; do not state from memory.

## Things you (as this specialist) would ADD to the playbook

- **A "data flow diagram" page** showing for each policy: signal source (M365 connector / MDE feed / Syslog from SWG) → MDA backend → action surface (Entra disable / Purview label apply / file move / SWG block-script export) → audit surface (`CloudAppEvents` row / Sentinel ingest / Graph Security API alert pull). Without this, the practitioner cannot trace cause → effect → evidence.
- **A "tenant integration prerequisites" checklist** that goes beyond licensing: (a) Entra provisioning enabled for each SaaS that will be SCIM-cascaded, with documented latency target; (b) Graph Security API app registration for non-Sentinel SIEM integration; (c) Sentinel workspace and Defender XDR connector enabled if Sentinel-bound; (d) Power Automate environment + Premium licence if governance-action flows in scope; (e) ServiceNow / Jira webhook receiver or Sentinel-to-ITSM connector deployed.
- **A "Graph throttling budget" appendix**: estimate file-policy scan rate per tenant size (50k seats × 5M OneDrive files = baseline scan duration estimate); document the Graph rate-limit headers and how MDA surfaces them in the activity log (it does not, well); name the Microsoft `Graph throttling` documentation URL.
- **A "SCIM cascade catalogue"** for the top 10 connected SaaS in a typical BFSI estate: for each, the disable signal latency, the per-app footguns (Salesforce Permission Set sticky on deactivation; Workday Worker vs Account distinction; AWS IAM Identity Center vs Organization SCIM divergence; ServiceNow Inactive vs Locked Out semantics).
- **An "Okta-as-primary-IdP" addendum**: the playbook is Entra-first throughout. For BFSI shops still on Okta with Entra as resource tenant, write the actual mechanics — Okta → Entra SAML federation as the upstream, the device-trust gap (Okta Verify state does not flow into Entra), the suspend-user signal flow (deactivate in Okta → SCIM to Entra → Entra propagates further), CAAC onboarding requiring app federation re-pointed at Entra.
- **A "DR / fail-mode matrix"** as a table: scenario (CAAC proxy region outage, MDA backend regional outage, Defender XDR outage, Entra outage, Sentinel ingest stall) × component (inline session policy, API-mode file policy, governance action queue, alert pipeline, SIEM forwarding) × behaviour (fail-open / fail-closed / queue / drop) × operator notification (yes / no / how).
- **A "policy-as-code workaround" section** since native PaC is absent: a documented Git workflow with portal JSON exports as the source of truth, a `diff` review process on policy changes, an alerting query that detects out-of-band portal edits via `AuditLogs`. This is what bigger MDA customers actually do.
- **A "regulator audit-trail join query" appendix** — the actual KQL for the three-way join across `SigninLogs` + `AuditLogs` + `CloudAppEvents` that an auditor will accept as evidence. Without runnable KQL, the playbook's audit-evidence claims are unverifiable.

## Overall verdict

- Playbook v0 fit for practitioner use as-is? **NO-WITH-CRITICAL-FIXES**
- One-line reason: Solid policy-by-policy turn-on guidance but treats MDA as a self-contained product — the actual integration surface (SIEM connector choice, Graph throttling, SCIM cascade direction, Okta-fronted CAAC mechanics, multi-geo coverage, DR fail modes, the wrong-by-omission session-correlation-ID join) is unwritten, and a BFSI practitioner following it will configure controls that look correct in the portal but produce empty SIEM joins, undetected coverage gaps, and orphan policy objects on the first audit cycle.
