# Senior cyber threat / risk specialist - playbook v0 review

Reviewer stance: senior threat / risk specialist, MITRE ATT&CK Enterprise & Cloud matrices, ATLAS for AI, adversary-tradecraft lens. Other lenses have already covered architecture, product UX, and GRC. This review is restricted to **threat coverage**: does each named policy actually counter the techniques the playbook implies, and where does the threat model leak.

Vendor-marketing terms used by the draft and rejected on sight: "highest-likelihood incident classes" (line 60 — unmeasured), "bypasses every network-layer control" (line 77 — overstated; depends on consent path), "the higher-risk class" (line 82 — true but used as an excuse not to map techniques).

## Highest-impact findings (rank-ordered)

1. **Zero MITRE ATT&CK mapping in a playbook whose entire value proposition is "what risk each switch actually reduces" (line 60).** Every policy says "Risk reduced: <prose>"; not one cites a TID. A practitioner cannot defend "this control covers technique X" to a purple team, an attestation, or a board. Minimum required mapping the v0 omits:
   - Policy 1 (Shadow IT + GenAI triage) → discovery / inventory of **T1567 Exfiltration Over Web Service** channels and **T1102 Web Service** C2 candidates. Not a *counter* on its own — a *visibility precondition*. The draft writes it as a risk-reducing control. It is not. It is a discovery feed.
   - Policy 2 (OAuth ban) → **T1528 Steal Application Access Token**, **T1550.001 Application Access Token**, and post-consent **T1567 / T1071.001** exfil. The draft only narrates the "consent-phish" path; does not name the technique.
   - Policy 3 (Mass-download alert) → **T1530 Data from Cloud Storage Object** + **T1213.002 SharePoint** + bulk-stage of **T1567.002 Exfiltration to Cloud Storage**. Detection only; not prevention. The draft does not distinguish.
   - Policy 4 (Impossible travel) → **T1078.004 Cloud Accounts** valid-credential use. Detection, low-fidelity, country-only — see finding 6 for why this is brittle.
   - Policy 5 (Block download to unmanaged) → defeats **T1530** via untrusted endpoint, partially defeats **T1567** when the exfil channel is the user-initiated download. Does NOT counter T1530 when the attacker uses a managed endpoint or token-replay.
   - Policy 6 (Auto-label PCI) → no ATT&CK *technique* countered directly; it is a precondition for downstream **encryption-at-rest** controls that frustrate **T1565.001 Stored Data Manipulation** and **T1486 Data Encrypted for Impact** (where Purview labels carry encryption). The draft frames it as exfil-prevention. It is not.
   - Policy 7 (Quarantine stale external shares) → **T1213 Data from Information Repositories** (long-tail). Detection + clean-up, not active counter.
   - Policy 8 (Auto-unsanction GenAI) → **T1567 Exfiltration Over Web Service** to AI endpoint; **ATLAS AML.T0010 ML Supply Chain Compromise** is *not* countered, only the egress channel is.
   - Policy 9 (Block paste to ChatGPT) → **T1052 Exfiltration Over Physical Medium** is the wrong mapping; the right one is **T1567 Exfiltration Over Web Service** narrowed to the clipboard sub-channel. The draft does not name either.
   - Policy 10 (Terminated user) → **T1078.004 Cloud Accounts** persistence, **T1136 Create Account** (where the SaaS-local account was created), and **T1098.001 Additional Cloud Credentials** when the user added a token before leaving.
   Without these mappings the playbook cannot be used in any threat-coverage matrix. **Mandatory before any "fit for practitioner use" claim.**

2. **Policy 2 (OAuth ban) is the worst overstatement in the playbook.** Line 77: "This bypasses every network-layer control because the traffic is app-to-Microsoft, not user-to-anywhere." That sentence is true for the *exfil* leg only. The *consent* leg is browser-initiated by the user and CAN be intercepted at the IdP (Entra consent governance), at the browser session (CAAC if onboarded), or at the network layer (SWG inspection of the consent screen). The playbook implies MDA-OAuth-ban is the primary control. It is not — it is the *post-consent* cleanup. The primary control is **Entra consent governance + admin consent workflow** (T1528 mitigation M1018). MDA OAuth-app page is detection + revoke of an event that should not have happened. Reframe.

3. **Policy 2 explicitly admits Application (client-credentials) grants are out of scope (line 82) and the playbook accepts this as a "residual gap" — but does not state that this is the technique class that matters most.** **T1098.001 Additional Cloud Credentials** (adding a secret or certificate to an existing service principal) and **T1136.003 Create Cloud Account** (creating a new service principal) are the dominant cloud-persistence techniques in real incidents (MIDNIGHT BLIZZARD / Storm-0558 / Storm-1283 patterns). MDA's OAuth page does not see them. The playbook should not be quiet about this — it should say "Policy 2 covers ~30% of OAuth-grant abuse; service-principal abuse is the other ~70% and needs **App Governance** (Entra) or **Defender for Cloud** workload-identity coverage." A practitioner reading v0 will think OAuth is solved. It is not.

4. **The threat model is silent on multi-stage / multi-channel adversaries.** The playbook implicitly assumes an attacker will exfil via one named channel (download, OAuth, paste-to-AI) at a rate that hits one of the rate-based detections. Realistic adversaries:
   - Stage data in OneDrive, share externally via anonymous link (defeats Policy 3 — no download; Policy 7 — fires 180 days late).
   - Exfil via **T1537 Transfer Data to Cloud Account** (account-to-account, attacker's own M365/Workspace tenant) — Policy 5 does not see this if both sides are sanctioned/federated SaaS.
   - Exfil via **T1048.002 Exfil Over Asymmetric Encrypted Non-C2 Protocol** by pasting a PGP-encrypted blob into the AI prompt or into a comment field (DCS cannot inspect the cleartext; `always_apply_if_unscannable: false` fails open per Policy 5).
   - Slow-and-low download: 100 files/day × 30 days defeats the 5,000-in-30-min threshold entirely (Policy 3) and is below most UEBA noise floors. The playbook offers no defence on the slow path.
   - Lateral via **T1199 Trusted Relationship** — partner-tenant guest access. The playbook does not address B2B guest exfiltration at all.

5. **Policy 8 (GenAI unsanction) is a thin enforcement chain dressed as a control.** Five sequential failure modes the playbook only partially names:
   - (a) **T1090.002 External Proxy** — attacker routes traffic through an approved AI domain that's actually a redirector (line 161 only addresses risk-score gaming, not domain-fronting).
   - (b) **T1568.002 Domain Generation Algorithms** + **T1102 Web Service** — modern GenAI vendors front on shared CDNs; the "Generate block script for SWG" produces FQDNs that don't match (line 163 acknowledges this, but doesn't say the technique class is *defeated*, not just "partially fails").
   - (c) Mobile / BYOD bypass — Network Protection is endpoint-only; phone hotspot defeats it entirely (**T1133 External Remote Services** from the attacker's lens, **T1078.004 Cloud Accounts** from the auditor's).
   - (d) Personal Microsoft Account or personal Google sign-in to ChatGPT — Entra never sees the session (line 318 of the architect lens noted the carve-out; v0 does not state it as a defeated control).
   - (e) Audio / multimodal — pasting a screenshot of the PCI table or recording a voice memo of customer data defeats Policy 9's clipboard-DLP inspection. DCS does not OCR or transcribe at the session layer.

6. **Impossible travel (Policy 4) is not a meaningful counter to T1078.004 in 2026.** The technique class — valid cloud account abuse — moved to **residential proxies / IPRoyal / 911-S5-class infrastructure** and **OAuth token replay (T1550.001)** years ago. Both defeat impossible-travel by construction: the attacker emerges from an IP geographically adjacent to the user. The playbook calls it a "compromised-account" detector (line 100). It catches the unsophisticated half. The playbook should state: *Impossible travel covers naive credential stuffing only; token replay and residential-proxy abuse defeat it.* The compensating control set is **Entra Token Protection (TP / SBM)**, **Continuous Access Evaluation (CAE)**, and **sign-in-risk policies** — none of which are MDA. State the gap.

7. **The playbook is silent on AI-specific threat classes (ATLAS) other than data leakage.** DSPM-for-AI is mentioned (line 23) as a feature, not a threat surface. ATLAS techniques the playbook does not cover and does not flag as residual:
   - **AML.T0051 LLM Prompt Injection** — a malicious document in OneDrive contains an indirect-prompt-injection payload; when a Copilot / agentic flow reads it, the agent is hijacked. MDA's file policies do not scan for prompt-injection patterns (no DCS classifier exists for them as of v0 publication). Once the agent has scopes equivalent to **T1098.001 Additional Cloud Credentials**, the attacker has an OAuth escalation path the playbook never considers.
   - **AML.T0052 Phishing** via AI-generated content — out of scope for MDA but should be named as residual.
   - **AML.T0010 ML Supply Chain Compromise** — when the "approved AI vendor" allowlist (Policy 8 key configuration) bakes in a vendor that is later compromised, the entire allowlist becomes an exfil channel. The playbook has no review cadence for the approved-AI list, no kill-switch posture, no statement that approving an AI vendor is a **third-party concentration-risk decision**, not a tag in the catalogue.
   - **Model inversion / membership inference via DSPM-for-AI** — DSPM stores prompts. Admin access to the prompt corpus is itself a high-value target (**T1213 Data from Information Repositories**, now targeting the AI-incident store). The playbook does not name DSPM-for-AI's incident metadata as a regulated content store that needs T1078/T1098 hardening on the admin role.
   - **Indirect prompt injection causing OAuth grant escalation** — a Copilot agent reads a poisoned email/file, follows an instruction to grant consent to a third-party app, and now the attacker has **T1528 Steal Application Access Token** *initiated by an internal service principal*. MDA's OAuth detection does not distinguish "user clicked Consent" from "AI agent invoked Consent on the user's behalf." Material gap.

8. **Policy 5 (block download of labelled files to unmanaged browsers) has a quiet T1550.001 (Application Access Token) bypass the playbook does not name.** If the attacker has the user's refresh token (harvested via infostealer, **T1555.003 Credentials from Web Browsers**), they re-issue from any endpoint and present a *managed* device claim if the token was originally issued on a managed device. The session policy fires on *current device tag*, not on token-binding state. The playbook does not state that Policy 5 must be paired with **Token Protection (Entra)** or it has a token-replay bypass. Realistic adversaries do this.

9. **Policy 3 (mass-download alert) ignores the entire native-sync exfiltration class.** OneDrive sync client + selective sync = bulk staging without ever generating a "download file" event in `CloudAppEvents` in the way the policy detects. The events are present but classed differently and the threshold (5,000 / 30 min) does not catch sync-driven staging. The playbook does not name **T1074.002 Remote Data Staging** via SaaS sync as a separate detection requirement.

10. **The playbook does not address the destructive-action threat class at all.** **T1485 Data Destruction**, **T1486 Data Encrypted for Impact** (ransomware-in-SaaS — encrypting OneDrive / SharePoint via the Graph API after **T1528** token theft), and **T1490 Inhibit System Recovery** (deleting retention policies / SharePoint version history via admin OAuth). MDA does not natively detect mass-delete or mass-encrypt events as separate templates. The Activity policy engine can be configured for it (Activity type = File deleted, threshold N) — the playbook should include a Day-30 policy "Mass-delete / mass-overwrite anomaly, alert + governance-action review" and does not. This is a 2024-2026 incident class (Storm-0501, BlackCat-on-SaaS variants) the playbook misses entirely.

11. **Terminated-user policy (Policy 10) misses the persistence techniques that actually matter.** The named scenario (line 183, "secondary local account in connected SaaS") is one path. The unaddressed paths:
   - **T1098.001 Additional Cloud Credentials** — terminated user added a service-principal secret or a personal access token (PAT in GitHub Enterprise, API key in Salesforce, app-password in Workspace) before leaving. MDA's terminated-user policy does not enumerate these. **Manual offboarding runbook** is the only mitigation and the playbook does not call for one.
   - **T1556 Modify Authentication Process** — the terminated user added an alternate MFA method or a backup email before leaving. MDA does not see this; Entra does.
   - **T1136.003 Create Cloud Account** — terminated user, while privileged, created a backdoor account in Salesforce / AWS / GitHub. MDA's anomaly looks for *activity* by the deleted Entra account, not for *new accounts* the deleted Entra account created.
   The playbook should reframe Policy 10 as "one of N offboarding-gap detections" and name the N.

12. **CAAC's `*.mcas.ms` reverse-proxy is itself a Tier-1 attacker target and the threat model is silent.** A compromise of the CAAC proxy, the certificate, or the URL-rewriting logic = **T1557 Adversary-in-the-Middle** against every protected session simultaneously. Microsoft is a Tier-1 cloud provider; the risk is low-probability. But the **blast radius** is total — the attacker would see plaintext for every CAAC-protected session in the affected geo. The playbook does not state this as a concentration risk under the CAAC architecture. (The architect lens flagged Microsoft-stack concentration; this is the sharper, technique-specific version of the same concern.)

13. **No threat-actor profile.** The playbook lists incident classes without naming the actor sets that drive them. A practitioner-grade playbook should state which TTP cluster each policy counters: e.g. Policy 2 (OAuth ban) primarily counters **financially-motivated consent-phishing** (TA453-like patterns) and **state-aligned identity-targeting** (MIDNIGHT BLIZZARD / Storm-0558 / Storm-2372 patterns); Policy 8 (GenAI unsanction) counters **insider negligence and second-stage exfil** but not **deliberate insider exfil with knowledge of the controls**. Without this, the SOC cannot prioritise alert triage.

14. **The "compromised-account" framing (line 100) is anachronistic for cloud-first attacks.** Modern intrusions begin with **T1566.002 Spearphishing Link → T1528 Steal Application Access Token → T1550.001 Application Access Token Replay**, NOT with stolen passwords. Impossible travel (Policy 4) catches password-stealers; it does not catch token-stealers (the token was issued to the legitimate user on the legitimate device; replay from anywhere produces no impossible-travel signal). The Day-1 set has zero defence against the dominant attack pattern of 2024-2026. State it.

15. **No ATT&CK coverage table.** The cross-vendor `synthesis/` directory should hold a coverage matrix per vendor (which tactics, which techniques, prevent vs detect vs respond, with-CAAC vs without-CAAC). The playbook does not even point to one. Without a coverage matrix the question "is MDA the right control for technique X?" has no answer in this document.

## Specific fixes the v1 must apply

- **[ALL POLICIES]** Add a row to each policy table: **`MITRE ATT&CK technique(s) countered`** with TID + name + Prevent/Detect/Respond verdict. Where the technique is only partially countered, say so (e.g. "T1530 — Detect only, post-event").
- **[POLICY 1, Shadow IT]** Reframe from "Risk reduced" to "Visibility precondition for". Discovery without enforcement does not reduce risk; it enables a control decision. The current framing is vendor marketing.
- **[POLICY 2, OAuth]** Add a critical-limitation row: "**T1098.001 Additional Cloud Credentials** and **T1136.003 Create Cloud Account** for service principals are NOT covered. App Governance (separately licensed) or Defender for Cloud is the compensating control. Service-principal compromise is the dominant cloud-persistence pattern; treat this as a P1 residual gap." Move from "critical limitation" footnote to top of the policy table.
- **[POLICY 2, OAuth]** Reposition as a "post-consent cleanup" control, not a primary defence. State that the primary control for **T1528** is Entra admin-consent workflow + verified-publisher restriction, configured in Entra not MDA.
- **[POLICY 3, mass-download]** Add: "Does NOT catch slow-and-low exfiltration (e.g. 100 files/day for 30 days). Add a parallel low-and-slow detector via Sentinel KQL on `CloudAppEvents` aggregating per-user weekly download volume against a 90-day baseline." Provide the KQL.
- **[POLICY 3, mass-download]** Add: "Does NOT catch native-sync staging (T1074.002 via OneDrive sync). The event class differs. Add a separate detection for sync-client bulk-pull patterns or rely on MDE endpoint telemetry."
- **[POLICY 4, impossible travel]** Add: "Defeated by **T1550.001 Application Access Token** replay (token issued to legitimate user, replayed from attacker infra) and by residential-proxy infrastructure that places the attacker IP near the user. Compensating controls: Entra Token Protection (preview/GA status to verify) + Continuous Access Evaluation + sign-in-risk policies. **MDA's impossible-travel is naive-credential-stuffing-only**."
- **[POLICY 5, block download]** Add a Token Replay row: "Bypassed by **T1550.001** when the attacker possesses a valid refresh token issued on a managed device. Pair with Entra Token Protection (binds token to device) or this control has a stealthy replay bypass."
- **[POLICY 5, block download]** State the `always_apply_if_unscannable: false` decision in ATT&CK terms: "Choosing false fails open on encrypted-by-user archives (T1027 Obfuscated Files or Information) and unscannable file types. An attacker who 7zips the payload with a password defeats this control. Document the residual."
- **[POLICY 6, auto-label PCI]** Re-frame: not an exfil control. It is a **labelling precondition** for downstream encryption (Purview label-applied encryption) and for **Policy 5's filter to even fire**. Without Policy 6, Policy 5's `Sensitivity label ∈ {Confidential}` matches nothing. State the dependency chain.
- **[POLICY 8, GenAI unsanction]** Add a "Defeated by" row enumerating: BYOD mobile / hotspot, personal MSA or Google sign-in, CDN-fronted vendor domains, image/audio bypass of clipboard inspection. Quantify each as a residual probability category, not just an aside.
- **[POLICY 8, GenAI unsanction]** Add a P0 control-design requirement: **the approved-AI allowlist is a concentration-risk decision and must have a documented review cadence (quarterly minimum), kill-switch posture, and supply-chain-compromise contingency**. The current "exclude by app ID" line treats it as configuration; it is governance.
- **[POLICY 9, paste to ChatGPT]** Add: "Defeated by upload of image / screenshot / audio of the same data (DCS does not OCR or transcribe at the session layer). Defeated by typing the data (no paste event)." State the residual as substantial.
- **[POLICY 10, terminated user]** Add three new sub-cases: (a) service-principal secrets / PATs / API keys added before termination → check Entra Audit Logs `AppRoleAssignment` and `Add service principal credentials`; (b) MFA-method additions or backup-email changes (T1556) before termination → Entra Audit `User registered alternative authentication device`; (c) accounts created by the terminated user in connected SaaS (T1136) → per-SaaS admin audit, not in MDA. Include a manual offboarding runbook reference.
- **[DAY 30 SECTION]** Add a new policy: **Mass-delete / mass-overwrite anomaly on M365** countering **T1485 Data Destruction** and **T1486 Data Encrypted for Impact** (ransomware-in-SaaS). Activity policy on `File deleted` and `File modified` events at high velocity per user, alert + manual governance review (NOT auto-suspend; see Policy 3 trap). This is a 2024-2026 incident class entirely absent from v0.
- **[CONCENTRATION RISK]** Reframe the existing concentration-risk paragraph (line 54) in technique terms: a CAAC proxy compromise = **T1557 Adversary-in-the-Middle at internet-scale**. Single Entra identity-plane compromise = **T1098.001 / T1556 / T1528** across the entire Microsoft stack simultaneously. Quantify the blast radius in technique terms, not just vendor-count terms.
- **[NEW SECTION]** Add an ATT&CK coverage matrix at the end (or link to one in `synthesis/`) covering: T1078.004, T1098.001, T1136.003, T1528, T1530, T1537, T1550.001, T1556, T1565.001, T1567, T1567.002, T1199, T1213, T1213.002, T1485, T1486, T1557, T1574, T1090.002, T1568, T1199 — with MDA's coverage verdict per technique (Prevent / Detect / Respond / Out of scope) and the compensating control where Out of scope.
- **[NEW SECTION]** Add an ATLAS coverage statement covering at minimum AML.T0010, AML.T0051 (prompt injection), AML.T0024 (model output extraction), AML.T0040 (Discover ML model ontology via DSPM access).

## Claims to verify or remove

- **Line 60**: "highest-likelihood incident classes" — unmeasured. Cite an incident dataset (M-Trends, DBIR, Mandiant FLARE quarterly) or strike.
- **Line 77**: "bypasses every network-layer control because the traffic is app-to-Microsoft, not user-to-anywhere." True for post-consent exfil traffic; false for the consent screen itself (which is browser-rendered and inspectable). Tighten.
- **Line 100**: "particularly password-spray / credential-stuffing scenarios" — concedes the policy is naive-attack-only. State that token-replay and residential-proxy classes are out of scope; do not leave the reader thinking impossible-travel covers them.
- **Line 158**: "Once leaked, the data is in the model." — overbroad. Training behaviour varies by tier (consumer vs Plus vs API vs Enterprise) and by vendor. State the actual training-on-input policy per tier (OpenAI Enterprise = no training by default; ChatGPT Free/Plus historically = yes-with-opt-out). Without this nuance the threat-reduction claim is vendor-FUD.
- **Line 164**: "Microsoft's risk score is opaque (90+ factors, weights not disclosed)" — correct on opaqueness, but the playbook then uses risk score ≤5 as a control trigger (line 161) without stating the FP/FN baseline. Either measure FP/FN in pilot and document, or drop "Risk score ≤5" as the trigger.
- **Line 192**: "Detects when an Entra-deleted account corresponds to a still-active account in a connected SaaS app." — depends on shared identifier (the trap is named but the *coverage rate* is not). State: "for SaaS apps where the email identifier does not match exactly, this control has 0% coverage." That is the honest framing.
- **Line 24**: "UEBA / anomaly detection — built-in policies for impossible travel, mass download, terminated-user activity in connected apps, suspicious OAuth grants." — UEBA framing implies behavioural baselining. These are mostly threshold + rule-based policies with a 7-day learning window (product lens finding). Stop calling them UEBA.

## Things you (as this specialist) would ADD to the playbook

- **An ATT&CK coverage table** (per finding 15 and the fix above). Mandatory for any threat-coverage claim.
- **An ATLAS coverage table for AI-specific threats**. Currently the playbook treats AI as a data-loss channel only; the AI surface has its own threat techniques.
- **A threat-actor profile per policy**: for each Day-1/30/90 control, state which named actor TTP clusters the policy meaningfully counters (e.g. "Policy 2 counters consent-phishing patterns associated with Storm-2372, Storm-0558 admin-consent abuse"; "Policy 8 counters insider negligence + opportunistic exfil; does NOT counter trained-adversary exfil with knowledge of the allowlist").
- **A "slow-and-low" detection layer** — Sentinel KQL queries over `CloudAppEvents` aggregated weekly/monthly per user against a baseline, covering download volume, OAuth-grant additions, external-share creation, MFA-method additions. The playbook is built around fast-and-loud thresholds and the slow-and-low channel is wide open.
- **A purple-team / atomic-test cadence**. Every named policy should have a periodic detection test using Atomic Red Team or PURPLE LABS techniques. Currently the playbook tells the reader to "monitor FP rate for 4 weeks" — no equivalent guidance for *true-positive* validation.
- **A destructive-actions detection set**. Mass-delete (T1485), mass-encrypt-via-Graph (T1486), retention-policy-deletion (T1490 cloud-flavour), version-history-purge. None of these are in v0.
- **A B2B / partner-tenant exfiltration policy**. T1199 Trusted Relationship via guest accounts is a real channel and the playbook does not address it. Activity policy: "external-domain guest user downloads / shares N+ files in T window" with alert.
- **An AI-agent control envelope**. As Copilot / agentic flows proliferate, "user clicked Consent" and "AI agent invoked Consent on behalf of user" must be distinguishable in audit. State the audit-event field that distinguishes them, or flag that they cannot currently be distinguished and the resulting **T1528-via-agent** is an undetected technique class.
- **A token-replay detection statement**. Pair every CAAC-protected policy with the Entra control that addresses T1550.001 (Token Protection / CAE), or state explicitly that the policy has a known token-replay bypass.
- **A residual-risks-in-ATT&CK-terms table**. The existing "Residual risks MDA does not address" table (line 192) is in product terms (BYOD-mobile, cert-pinning, etc.). Rewrite it in technique terms — readers can then map to their own coverage matrix instead of re-deriving.

## Overall verdict

- **Playbook v0 fit for practitioner use as-is? NO-WITH-CRITICAL-FIXES**
- **One-line reason:** Strong on operational footguns and limitations but zero MITRE ATT&CK mapping, silent on token-replay / service-principal persistence / destructive-actions-in-SaaS / B2B exfil / AI-agent-mediated consent abuse, and overstates Policy 2 (OAuth) and Policy 4 (impossible travel) coverage against the techniques that actually dominate cloud incidents in 2024-2026; needs a threat-coverage rewrite, not just an edit pass.
